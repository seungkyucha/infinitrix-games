## Patterns to Avoid
- **[Cycle 1]** When copying assets from a generic template, leaving unused assets behind causes unnecessary network requests (3 leftover: player.svg, enemy.svg, etc.)
- **[Cycle 1]** Establishing a "0 external assets" principle but then loading Google Fonts — mismatch between spec principles and implementation. If you set a principle, verify it with a checklist at the code review stage
- **[Cycle 1]** `confirm()`/`alert()` are unusable in iframe environments — in-game confirmation UI must always be implemented as Canvas-based modals
- **[Cycle 1]** If you just put a setTimeout lock intending to "add animation later," it ends up never being implemented. Movement animation is core feedback and must be implemented from the start
- **[Cycle 2]** Even if you specify input mode branching behavior in the spec, if you only declare the condition variable in code but don't actually use it in branching, it behaves differently from intended. Post-implementation spec cross-reference checklist is mandatory
- **[Cycle 2]** In core utilities like TweenManager, paths where `cancelAll()` and `update()` can be called simultaneously must be blocked at design time. Runtime collection modification should be protected with a deferred pattern
- **[Cycle 2]** **If you don't create a matrix of which systems to update in every game state machine state, bugs occur where core systems like tween/physics stop in certain states.** B1, where tw.update() wasn't called in CONFIRM_MODAL leaving the modal fixed at alpha=0, is the representative case. Include state × system matrix as a mandatory spec item
- **[Cycle 2]** If you save (saveBest) then compare (getBest), it's always false — always maintain "judge first, save later" order for state changes and judgments (B4)
- **[Cycle 2]** The pattern of delaying game state transitions with setTimeout was already a problem in Cycle 1 but recurred in Cycle 2 (B3). Specify the rule of completely replacing with tween onComplete callbacks in coding guidelines
- **[Cycle 3]** When using tween-based delayed transitions (state change via onComplete), if the original condition evaluates true every frame during the delay period, the callback gets called repeatedly. **Always set a guard flag (`waveClearing = true`) to ensure single execution.** (B1: waveComplete repeated calls causing infinite gold/score accumulation)
- **[Cycle 3]** When multiple state transition tweens are simultaneously active, the later-completing tween overwrites the earlier-completed transition. **A priority concept is needed for state transitions** — e.g.: GAMEOVER transition always takes priority over WAVE_PREP. Place pre-checks like `if (lives <= 0) return;` at the entry of transition functions. (B2)
- **[Cycle 3]** Even specifying "no SVG filters" in the spec, feGaussianBlur recurs at the asset creation stage (Cycle 2 B5 → Cycle 3 B4). Spec principles are only effective when automated verification scripts are applied to asset files, not just code
- **[Cycle 3]** "Ghost code" where variables are declared but never actually updated/used keeps recurring (Cycle 2: input mode branching, Cycle 3: livesAtWaveStart). **Include verification that declared variables are used per spec intent in code review checklist.**
- **[Cycle 4]** If `add()` is called immediately after TweenManager's deferred `cancelAll()`, the `_pendingCancel` flag deletes new tweens as well. **The race condition between cancelAll and add is a blind spot of the deferred pattern.** Separate an `clearImmediate()` immediate cleanup API, or flush with `_pendingCancel = false` + `_tweens.length = 0` right after cancelAll. (B1: Game start impossible CRITICAL)
- **[Cycle 4]** SVG asset recurrence has been 3 consecutive cycles (Cycle 2→3→4). It's definitively proven that "specifying ban in spec" alone cannot prevent asset creation stage violations. **Cannot be resolved without automated grep verification scripts.**
- **[Cycle 5]** Even after introducing automated verification scripts, running them only after code completion means residual patterns (ASSET_MAP, SPRITES, preloadAssets) are found in 1st round review. **Verification must be run frequently during coding to be effective.** Running once after code completion is insufficient.
- **[Cycle 5]** When both tween updates and direct assignment exist simultaneously for a single variable (bpm), dual registration problems occur. **Update paths for a single value must be unified to exactly one.** (1st round B2: bpmObj tween + bpm direct assignment duplication)
- **[Cycle 5]** When copying the assets/ directory from a generic template, SVG files unreferenced in code keep remaining. Code-level verification alone cannot prevent file remnants, so **asset directory auto-cleanup or unreferenced asset warning scripts are needed.**
- **[Cycle 5]** Even specifying "all state transitions must use beginTransition()" in the spec, direct transitions like startGame()/goToTitle() occur. **Spec principle vs implementation mismatches must be explicitly verified in code review checklist.** Even without functional issues, missing transition animations degrade UX.
- **[Cycle 6]** Despite planning 3-stage auto-verification (before/during/after coding), assets/ directory remaining was found in 1st round review. **Verification scripts are insufficient when merely specified in specs — they must be integrated into CI/build pipeline with forced execution to be effective.**
- **[Cycle 6]** Utility functions that directly depend on global objects (`speed()` directly accessing `ball`) are untestable. **Game logic functions should be written as pure function patterns receiving data through parameters to enable unit testing.**
- **[Cycle 7]** assets/ directory + SVG asset recurrence has occurred in most of 7 cycles. Found in 1st round review even after "termination" declarations in Cycles 5~6 (deleted in 2nd round). **Spec specification, auto-verification scripts, and 3-stage checklists all have limited effectiveness** confirmed. Only a forced mechanism that fails the build when `assets/` exists in CI/build hooks is the fundamental solution.
- **[Cycle 7]** Despite specifying pure function design principles in the spec and mostly following them, 1st round review found 5 combat/collision functions with direct global P references (all parameterized in 2nd round). **When principles are established, a comprehensive verification checklist must verify "all" functions.**
- **[Cycle 7]** Mismatch between spec values (critical Lv3=30%) and implementation values (25%). **Spec-implementation accuracy verification must go down to the numerical level, not just feature existence.**
- **[Cycle 8]** assets/ directory recurrence for 8 consecutive cycles. Pre-commit hook scripts were even specified in spec §13.1 but not actually registered to the pipeline, making them ineffective. **Writing "create a script" in the spec and actually registering it in `.git/hooks/` are completely different steps.** Next cycle must include hook registration itself as a verification item.
- **[Cycle 8]** Even with the `beginTransition()` bypass principle established, direct `enterState()` calls occur for states like PAUSED where "immediate transition is natural." **A design decision is needed on whether to allow exceptions to the principle, or to add an immediate transition mode to beginTransition().**
- **[Cycle 10]** "Fix regression" pattern where new bugs are introduced during fix process. While fixing 4 1st-round MINOR issues, `render()` signature change without passing `timestamp` caused new CRITICAL in 2nd round. **After fixes, full-flow regression testing (TITLE→MAP→BATTLE→VICTORY/DEFEAT) must be performed.** Individual fix verification alone is insufficient.
- **[Cycle 10]** Spec game-id (`mini-card-battler`) mismatched with actual deployed directory name (`mini-card-crawler`). When naming changes during spec→implementation, the spec must also be updated to maintain traceability.
- **[Cycle 10]** Even with 0 asset references in code, if the assets/ directory physically exists it's a spec violation. **Code-level clean ≠ filesystem-level clean.** Searching for `fetch`, `Image()`, `assets` strings cannot detect unnecessary file remnants. Pre-commit hooks must block directory existence itself.
- **[Cycle 10]** Inconsistent extreme abbreviation (`shopI` vs `shI`) caused CRITICAL crash (ReferenceError on shop entry). **When abbreviating for code compression, consistency between variable declarations and references within functions must be verified.** 'use strict' mode catches undeclared variables at runtime, but can't prevent when declared with a different name. Pre-defining abbreviation naming rules (prefix patterns etc.) is safer.
- **[Cycle 11]** `let gridCache` declaration positioned after `resizeCanvas()` call caused TDZ (Temporal Dead Zone) crash → game completely non-functional. **`let`/`const` variables must always be declared before first use (including function calls).** A basic mistake of variable declaration order can lead to CRITICAL, so initialization code block execution order must be explicitly verified.
- **[Cycle 11]** In idle games, not calling `updateProduction()` during tab switching (prestige/stats) stops production. **In idle/incremental genres, the production system must always run regardless of UI state.** Classify "background systems" separately in the state × system matrix.

## Genre-Specific Know-how
- **Puzzle [Cycle 1]**: 5×5 grid + slide merge mechanics have high fun-to-implementation-difficulty ratio. Key is the visual reward of "merge to evolve." `merged[][]` array preventing duplicate merges per turn is an essential technique for 2048-style games. Dynamic difficulty (score-based block distribution change) creates natural tension escalation without explicit difficulty selection.
- **Arcade/Shooting [Cycle 2]**: Key to vertical scroll shooters is "instant hit satisfaction" and "gradual power fantasy." Screen shake + explosion particles + score popup trio creates per-shot satisfaction. Wave system works better with formula-based scaling like `spawnInterval = max(400, 1800 - wave × 60)` than manual tables for maintainability. Setting player hitbox smaller (24×24) than visual (40×48) reduces "unlucky deaths" for appropriate perceived difficulty. Boss with 3-phase cycle + HP-based acceleration creates deep combat with minimal code. **Caution:** When game states grow to 6+ types, it's easy to forget which systems (tween, physics) should run in each — matrix is mandatory.
- **Strategy/Tower Defense [Cycle 3]**: Core TD fun is "puzzle-like satisfaction of finding optimal placement with limited resources" and "power growth fantasy." The trinity of 3 tower types (fast single/area slow/explosive AoE) creates deep strategy with few types. S-shaped fixed path + First targeting (waypoint index + progress) is simple to implement yet intuitive. Wave scaling formulas (`min(20, 5+N×1.5)` enemy count, `max(500, 2000-N×80)` spawn interval) are more maintainable than manual tables. **Caution:** TD has more in-loop systems (wave manager, enemy movement, tower attack, projectiles, particles, economy) than shooters, making state transition race conditions more likely. Always use guard flags for tween delayed transitions and specify transition priorities (GAMEOVER > WAVE_PREP). Dynamic balance correction (crisis bonus, skilled player challenge) is a good idea but must include actual operation verification in testing.
- **Casual/Endless Runner [Cycle 4]**: Core runner fun is "one more game" addictiveness and "instant reaction satisfaction." 3-lane + jump simple controls lower entry barrier, distance-proportional speed increase (`min(600, 200 + distance × 0.04)`) creates natural difficulty curve. Procedural chunk generation (6 weighted patterns + safe lane guarantee) enables replay value through reaction testing rather than pattern memorization.
