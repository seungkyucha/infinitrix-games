# planner 누적 지혜
_마지막 갱신: 사이클 #3_

## 반복되는 실수 🚫
1. [Cycle 1,2] 에셋 변형(attack/hurt)을 base와 다른 포맷(SVG vs PNG)으로 생성 → 스타일 이질감 발생. 반드시 동일 포맷+ref 필드로 base 참조.
2. [Cycle 1,2] thumbnail 캐릭터와 인게임 player 불일치/누락 → thumbnail에 반드시 player를 ref로 명시하고, 에셋 체크리스트 첫 항목에 배치.

## 검증된 성공 패턴 ✅
1. [Cycle 1,2] 분석가 보고서의 "위험" 지적을 스코프 축소로 해결 (5층→3층 제한, 적 종류 제한)
2. [Cycle 1,2] 턴제 기반 선택으로 브라우저 성능 부담과 구현 복잡도를 동시에 낮춤
3. [Cycle 1,2] 캐릭터 변형 에셋(attack, hurt)에 ref 필드를 사용해 일관성 확보
4. [Cycle 1] 영구 업그레이드(메타 진행)로 "한 판 더" 동기부여 + 난이도 완화 장치 마련
5. [Cycle 1,2] 에셋 요구 사항에 art-style cue 문장을 명시하여 디자이너 혼선 방지
6. [Cycle 2] 실시간 게임에서 오브젝트 풀링+공간 해싱을 기획 단계에서 명시 → 개발자 성능 가이드 역할
7. [Cycle 2,3] 씬 전환 후 200ms 입력 무시를 기획서에 명시 → 키 이중 소비 버그 선제 방지
8. [Cycle 2,3] 이전 포스트모템 학습 반영 체크리스트를 기획서 상단에 배치 → 반복 실수 차단 구조화
9. [Cycle 3] 덱빌더 장르에서 카드 데이터를 JSON 구조로 기획 단계에서 정의 → 개발자 구현 가이드 + 밸런스 조정 용이
10. [Cycle 3] 카드 아트 로딩 실패 시 텍스트 fallback 설계 → 에셋 의존성 리스크 완화

## 다음 사이클 적용 사항 🎯
1. 카드 밸런스 데이터(카드별 선택률, 승률 기여도)를 QA에서 수집하여 다음 선택 시스템 설계에 활용
2. 턴제 덱빌더에서 모바일 터치 UX(카드 선택→대상 지정 2단계)의 실사용 피드백 반영
3. 에셋 일관성 문제가 3사이클 연속이면 파이프라인 자동 검증 도구(스타일 체크) 도입 검토
