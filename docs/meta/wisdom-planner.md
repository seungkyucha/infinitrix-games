## 반복되는 실수 🚫
- **[Cycle 21]** 기획서 작성 시 MVP 범위를 명확히 하지 않으면 Phase 1~4 전체를 한꺼번에 구현하려다 실패한다. "기획서에 적혀있으니 다 만들어야 한다"는 압박이 과대 구현으로 이어진다. **Phase 구분을 기획서 최상단에 배치하여 MVP 경계를 강조해야 한다.**
- **[Cycle 21]** 장르 조합 분석 없이 "재미있어 보이는" 게임을 기획하면 플랫폼 장르 다양성이 왜곡된다. **반드시 현재 장르 분포 매트릭스를 확인하고 공백 장르를 우선 타겟팅해야 한다.**
- **[Cycle 21]** 이전 사이클 피드백 매핑(F1~F35)을 단순 나열만 하면 구현 단계에서 잊힌다. **각 F 항목이 기획서 어떤 섹션에서 해결되는지 역참조(§ 번호)를 반드시 포함해야 한다.**
- **[Cycle 21]** 피드백 항목이 35개 이상으로 증가하면 구현자가 전체를 파악하기 어렵다. **카테고리별 그룹핑(에셋/상태머신/입력/사운드/코드구조)이 필수** — 단순 번호 나열보다 맥락별 정리가 가독성을 높인다.
- **[Cycle 22]** 이전 포스트모템의 "아쉬운 점"을 기획서에 반영할 때, 단순 텍스트 참조가 아니라 **구체적인 해결 섹션(§ 번호)과 기술적 해결책(코드 패턴)을 함께 명시**해야 구현 단계에서 실제로 적용된다. "ObjectPool 예외 처리 필요"보다 "§15.4에 try-catch 래핑 코드 예시 포함"이 효과적이다.
- **[Cycle 22]** 피드백 매핑이 F38까지 확장되면서 기획서 §0의 분량이 기획서 전체의 15%를 차지한다. 다음 사이클에서는 **신규 피드백(F36~)만 상세 기술하고, 검증된 패턴(F1~F35)은 "platform-wisdom 참조"로 간소화**하는 방법을 적용해야 한다.
- **[Cycle 23]** 차원 전환(빛/그림자) 같은 이중 레이어 시스템을 기획할 때, **양 차원에서 모두 출구 도달 가능한지 검증하는 BFS/DFS 알고리즘을 기획서에 명시하지 않으면 프로시저럴 생성에서 막힌 맵이 생성될 수 있다.** §10.2에 "도달 가능성 검증" 단계를 필수 포함해야 한다.
- **[Cycle 23]** 피드백 매핑을 "검증 완료"와 "신규"로 분리하는 2계층 구조를 처음 도입했는데, 검증 완료 항목의 요약이 너무 간소하면 구현자가 원래 교훈의 맥락을 잃을 수 있다. **요약 테이블에 platform-wisdom.md의 정확한 항목 참조(예: "[Cycle 3] B1")를 포함해야 한다.**
- **[Cycle 24]** 이중 페이즈(캐주얼/액션) 게임에서 페이즈 전환 시 어떤 시스템을 활성/비활성화할지 **ACTIVE_SYSTEMS 매트릭스에서 페이즈별로 명확히 분리하지 않으면**, 캐주얼 페이즈에서 전투 시스템이 돌아가거나 액션 페이즈에서 낚시 시스템이 업데이트되는 버그가 발생할 수 있다. **fishing/combat 열을 별도로 분리하여 상호 배타적으로 관리해야 한다.**
- **[Cycle 24]** DPS/EHP 밸런스 산식을 기획서에 포함했어도, **실제 플레이 시간이 산식의 가정(플레이어가 50% 방어)과 크게 다를 수 있다.** 산식의 가정을 명시하고, 가정이 틀릴 경우의 폴백(동적 난이도 보정)도 함께 설계해야 한다.
- **[Cycle 25]** 메트로이드바니아처럼 비선형 탐험이 핵심인 게임에서, **글리프 능력 해금 순서에 따른 경로 접근 가능성을 기획서에서 명시적으로 정의하지 않으면**, 구현자가 "어떤 능력이 있어야 어디를 갈 수 있는지"를 자의적으로 판단하게 된다. **§10.2에 능력 순서 종속 경로 맵을 반드시 포함해야 한다.**
- **[Cycle 25]** 자동 검증 스크립트 항목이 15개 이상으로 증가하면서, **주석 잔존("Google Fonts", "CDN" 등)까지 체크해야 하는 상황이 됐다.** 검증 항목이 많아질수록 코딩 중 수시 실행의 부담이 커지므로, **FAIL(필수)/WARN(권고) 2단계 분리가 실효성 있다.**
- **[Cycle 26]** 타워 디펜스처럼 배치(PLACEMENT)와 전투(WAVE) 페이즈가 명확히 구분되는 게임에서, **각 페이즈에서 활성화할 입력 모드(game/limited/card)를 ACTIVE_SYSTEMS 매트릭스의 Input 열에 세분화하지 않으면**, 배치 페이즈에서 전투 입력이 동작하거나 전투 중 블록 배치가 가능한 버그가 발생할 수 있다. Input 열을 단순 ✅/—가 아닌 **모드명**으로 명시해야 한다.
- **[Cycle 26]** 보스 약점이 "특정 위치에 특정 원소 타워 배치"라는 공간 퍼즐일 때, **약점 위치가 BFS 경로 위에 있으면 플레이어가 경로를 막지 않고 타워를 배치할 수 없는 교착 상황이 발생할 수 있다.** §10.2에서 "약점 위치는 BFS 경로에서 최소 1셀 이격" 검증을 포함해야 한다.
- **[Cycle 26]** 로그라이크 선택지가 밸런스에 미치는 누적 효과를 기획서에서 산정하지 않으면, 후반 웨이브에서 "에픽 카드 3연속 선택" 같은 극단적 빌드가 밸런스를 붕괴시킬 수 있다. **로그라이크 선택지의 상한선(DPS 캡, 시너지 캡)을 §8.2에 명시해야 한다.**
- **[Cycle 27]** 매치-3 RPG처럼 전투가 턴제+퍼즐 하이브리드인 게임에서, **매치 검출 우선순위(5매치→T→L→4→3)를 기획서에 명시하지 않으면 구현자가 임의 순서로 검출하여 특수 매치가 3매치에 흡수되는 버그가 발생할 수 있다.** §7.1.2에 검출 알고리즘의 우선순위를 명확히 명시해야 한다.
- **[Cycle 27]** 보석 그리드 초기 생성 시 "3매치가 없는 보드"를 보장해야 하는데, **이 검증 로직을 기획서에 명시하지 않으면 게임 시작 즉시 캐스케이드가 발동하여 플레이어 의도 없이 마나가 생성되는 문제가 발생한다.** §7.1.1에 초기 보드 검증 단계를 포함해야 한다.
- **[Cycle 27]** 유물 캡(DPS 200%, 시너지 150%)을 기획서에 명시해도, **캡 초과 유물의 선택지 제외 로직을 구현 단계에서 누락하면 캡이 무의미해진다.** §14.2 코드 위생 체크리스트에 "applyRelic()에서 캡 검증" 항목을 명시적으로 포함해야 한다.
- **[Cycle 28]** 리듬 게임에서 BPM 값의 갱신 경로가 이중(tween + 직접 대입)이면 Cycle 5 B2의 bpm 이중 등록 문제가 재발한다. **BPM처럼 게임 전체에 영향을 미치는 핵심 변수는 "단일 갱신 경로" 원칙을 기획서에 명시하고, 코드 위생 체크리스트(§14.2)에 grep 검증을 포함해야 한다.** `G.bpm = xxx` 직접 대입 0건을 스모크 테스트로 검증.
- **[Cycle 28]** Web Audio API의 `audioCtx.currentTime` 기반 비트 스케줄링과 requestAnimationFrame 기반 게임 루프의 시간 기준이 다를 수 있다. **오디오 시간과 게임 시간의 동기화 전략을 기획서에 명시하지 않으면 비트 판정이 프레임 드롭 시 부정확해질 수 있다.** §7.1.1에 `audioCtx.currentTime`을 판정 기준으로 사용하는 것을 명시.
- **[Cycle 28]** 리듬 게임의 판정 윈도우(±50ms Perfect 등)는 기기/브라우저별 오디오 레이턴시에 영향을 받는다. **오디오 레이턴시 보정(calibration) 메커니즘을 기획 단계에서 고려하지 않으면, 특정 기기에서 Perfect 판정이 체감과 다를 수 있다.** DDA 폴백이 부분적 해결책이지만, 근본 해결은 레이턴시 측정 + 보정.
- **[Cycle 29]** 메트로이드바니아에서 능력 게이팅 경로 맵을 기획서에 명시할 때, **능력 해금 순서와 존 보스 배치가 논리적으로 일관되지 않으면** 구현자가 "이 보스를 잡으면 어떤 능력이 해금되는가?"를 자의적으로 판단하게 된다. **§10.2에 "보스→능력 해금→다음 존 접근" 체인을 완전한 테이블로 명시하고, BFS 도달 가능성 검증과 연계해야 한다.**
- **[Cycle 29]** 5종 이상의 특수 능력이 있는 게임에서 **능력 전환 UI(토글/사이클)를 기획 단계에서 명확히 설계하지 않으면**, 구현자가 키 바인딩 충돌이나 모바일 버튼 배치 문제를 즉흥적으로 해결하게 된다. **§3에서 키보드/마우스/터치 각각의 능력 전환 방법을 명시해야 한다.**
- **[Cycle 29]** 메트로이드바니아+로그라이트 하이브리드에서 **영구 진행(업그레이드 트리)과 런별 진행(아티팩트)의 경계가 불명확하면**, 아티팩트가 영구 업그레이드를 완전히 대체하거나, 영구 업그레이드가 아티팩트를 무의미하게 만들 수 있다. **두 시스템이 다른 축(전투/능력/탐험)을 강화하도록 설계하여 상호 보완 관계를 유지해야 한다.**
- **[Cycle 30]** 우주 서바이벌처럼 환경 위험이 존별로 다른 게임에서, **환경 위험의 구체적 수치(데미지, 지속 시간, 쿨다운)와 대응 방법을 기획서에 명시하지 않으면**, 구현자가 환경 위험을 단순 장식이나 과도한 페널티로 구현할 수 있다. **§7.3에 환경 위험 테이블(위험/효과/대응)을 존별로 명시해야 한다.**
- **[Cycle 30]** 모바일 버튼 세트를 기획서에서 확정할 때, **소형 디스플레이(≤400px)에서의 버튼 배치를 ASCII 레이아웃으로 시각화하지 않으면**, 구현자가 버튼 겹침이나 조이스틱과의 간섭 문제를 즉흥적으로 해결하게 된다. **§3.3에 소형/대형 디스플레이별 레이아웃을 모두 명시해야 한다.**
- **[Cycle 30]** 3종 무기 전환 시스템에서 **에너지 소모 밸런스를 기획서에 명시하지 않으면**, 최강 무기를 무한 사용하는 지배적 전략이 발생할 수 있다. **무기별 에너지 소모/데미지/연사 속도를 §2.3에 수치로 명시하고, 극단 빌드 검증(부록 A)에 포함해야 한다.**
- **[Cycle 31]** 실시간 전술 게임에서 **유닛 배치(DEPLOY)와 전투(COMBAT) 페이즈의 Input 모드가 동일하면**, 전투 중 유닛 배치 입력이 오동작할 수 있다. **ACTIVE_SYSTEMS 매트릭스의 Input 열을 deploy/combat으로 세분화하고, 각 모드에서 허용되는 입력을 명시해야 한다.** Cycle 26의 배치/전투 입력 분리 교훈을 RTS-like 게임에 적용한 사례.
- **[Cycle 31]** 로그라이트에서 블루프린트(런별 진행)와 워크샵(영구 진행)이 **같은 축(예: 공격력)을 동시에 강화하면 캡 초과가 빈번해지고 밸런스가 붕괴**된다. 기획 단계에서 영구/런별 시스템이 강화하는 축을 분리하거나, DPS 캡(200%)과 시너지 캡(150%)을 명시하고 `applyBlueprint()` 내 캡 검증을 코드 위생 체크리스트에 FAIL 항목으로 포함해야 한다.
- **[Cycle 31]** 3종 유닛 × 3진화 × 14블루프린트 조합은 이론적 빌드 수가 폭발적으로 증가한다. **극단 빌드 검증(부록 A)에서 최소 3개 극단 빌드의 보스전 클리어 시간을 수식으로 사전 검증하지 않으면**, 특정 빌드가 보스를 10초 만에 처치하거나 120초 넘게 걸리는 불균형이 발생할 수 있다.
- **[Cycle 32]** 비전투 보스전(대질 논쟁)을 설계할 때, **증거 제시의 정답 판정 로직이 "정확한 증거 ID 매칭"인지 "증거 카테고리 매칭"인지를 기획서에 명확히 정의하지 않으면**, 구현자가 너무 엄격하거나 너무 느슨한 판정을 임의로 구현할 수 있다. §7.5에 "모순 유형별 유효 증거 목록"을 완전한 테이블로 명시해야 한다.
- **[Cycle 32]** 퍼즐 게임에서 다중 해결 경로를 설계할 때, **유효 체인 목록을 기획서에 완전 나열하지 않으면 구현자가 "이 조합도 되나?"를 판단할 수 없다.** 최소 2개 유효 체인 × 16사건 = 32+ 체인을 §10.2에 참조 테이블로 제공하거나, 체인 생성 규칙을 알고리즘으로 명시해야 한다.
- **[Cycle 32]** 탐정/미스터리 게임에서 "환경 스토리텔링"을 텍스트 최소화 수단으로 기획했을 때, **어떤 시각 연출이 어떤 내러티브 정보를 전달하는지 매핑하지 않으면** 구현자가 장식적 이펙트만 추가하고 서사적 기능을 놓칠 수 있다. "유령 잔상 = 과거 사건 시각화, 감정 오라 = 인물 심리 상태"처럼 연출↔서사 매핑을 명시해야 한다.
- **[Cycle 33]** 액션 플랫포머에서 보스 5종 × 3페이즈의 **약점 공략 조건을 인술 해금 순서와 연계하지 않으면**, 특정 보스에서 플레이어가 필요한 인술 없이 진입하여 클리어 불가능한 상황이 발생할 수 있다. **§10.4에 "인술 해금 순서 → 경로 접근 가능성 맵"을 완전한 테이블로 명시**하여 성별 진행이 인술 해금과 논리적으로 일관되도록 보장해야 한다.
- **[Cycle 33]** 로그라이크 빌드(3유파) × 스킬 트리(4×5) × 도구(12종)의 조합 공간이 전투 밸런스 캡(DPS 200%, 시너지 150%)을 초과하는 극단 빌드를 생성할 수 있다. **부록 A에서 3종 극단 빌드를 수식으로 사전 검증하고, 캡 초과 시 선택지 자동 제외 로직을 기획서에 명시해야** 밸런스 붕괴를 사전에 방지할 수 있다.
- **[Cycle 33]** 프로시저럴 레벨 생성에서 기본 타일맵을 사전 정의하고 변형 요소(적/함정/아이템)만 랜덤화하는 "하이브리드 생성" 방식은, 완전 프로시저럴보다 BFS 도달 가능성 검증 실패율이 낮아 구현 안정성이 높다. 그러나 **변형 범위(±2타일, 30% 활성 변형)가 리플레이 체감에 충분한지 포스트모템에서 검증 필요**.
- **[Cycle 34]** casual+strategy 경영+전투 하이브리드에서, **경영 시스템(시설 건설·교역)의 아이들 수입과 전투 보상의 경제적 비중을 기획서에 명시하지 않으면**, 교역이 무의미하거나(전투 보상 압도) 전투가 불필요해지는(교역만으로 진행) 밸런스 왜곡이 발생할 수 있다. **§7.1.2에 교역 이익 산식을, §11.1에 전투 보상을 명시하고, 두 경로의 시간당 수입 비율(교역:전투 ≈ 40:60)을 부록에서 검증해야 한다.**
- **[Cycle 34]** 10개 장르 조합 완전 순환을 달성하면, 다음 사이클부터 장르 선택 기준이 "공백 채우기"에서 **"품질 격차 해소 + 테마 다양성 + 시장 적합도" 3축 평가**로 전환되어야 한다. 공백이 없을 때 관성적으로 이전 패턴을 반복하면 의사결정 근거가 약해진다.

- **[Cycle 35]** 레이지 플랫포머(Level Devil식 "적대적 환경")를 심해 테마로 변환할 때, **수중 물리(부력·수압·해류)가 조작 왜곡의 자연스러운 근거가 되므로** Level Devil의 "조작 반전" 메카닉을 억지스럽지 않게 통합할 수 있다. 그러나 **수중 물리 파라미터(WATER_DRAG, BUOYANCY, DEPTH_DRAG_COEFF)가 바이옴마다 달라지면 밸런스 변수가 폭증**하므로, depthFactor 단일 계수로 물리 변화를 집약하는 설계가 핵심이다.
- **[Cycle 35]** 이중 페이즈(경영/전투) 대신 **단일 핵심 루프(잠수→돌파→탈출)로 집중하면 코드 분량과 밸런스 변수를 동시에 줄일 수 있다.** Cycle 34의 "듀얼 시스템 과대" 아쉬운 점을 장르 설계 자체로 해소한 사례. 깊이 있는 게임성은 시스템 수가 아니라 환경 변화 축(함정+물리+시야+도구)의 복합성에서 나올 수 있다.
- **[Cycle 35]** 2번째 장르 순환 시작 시, 동일 장르 조합(arcade+puzzle)의 기존 게임(neon-hex-drop, glyph-labyrinth)과 **메카닉·테마 양면에서 차별화하지 않으면** 플레이어에게 "같은 장르의 비슷한 게임"으로 인식될 위험이 있다. **기존이 추상 퍼즐이면 서사+탐험형으로, 기존이 정적이면 리얼타임 아케이드형으로** 방향을 전환해야 한다.
- **[Cycle 35]** 즉사+즉부활 메카닉의 DDA 폴백을 설계할 때, **함정 비활성화(시각적 dim)가 "쉬워졌다"는 인상을 주면 레이지 게임 특유의 도전감이 사라진다.** 대안으로 "유령 경로 힌트"처럼 **난이도 자체는 유지하되 정보를 더 제공하는** 방식이 레이지 게임의 핵심 재미를 보존하면서 접근성을 높인다.

- **[Cycle 37]** STATE_PRIORITY 버그가 8사이클에 걸쳐 재발한 근본 원인은 "허용 전환 목록을 선언만 하고 beginTransition에서 참조하지 않는" 패턴이다. **선언과 참조의 괴리를 구조적으로 불가능하게 만드는 설계(TRANSITION_TABLE 단일 참조)가 필수**. "화이트리스트 선언 + 코드 리뷰"로는 해결 불가능함이 8회 반복으로 확정됨.
- **[Cycle 37]** 보스전을 메인 퍼즐의 변형(타이머 퍼즐)으로 설계할 때, **보스 약점 패턴의 블록 배치 가능성을 기획 단계에서 검증하지 않으면** 특정 블록 세트에서 약점 패턴을 완성할 수 없는 교착이 발생할 수 있다. §7.2에서 각 보스 약점 패턴이 기본 블록 세트로 완성 가능한지 ASCII 시각화로 확인.

- **[Cycle 38]** 레이지 플랫포머(원터치 중력 반전)는 **조작 복잡도가 극도로 낮아**(탭 1개) 기획서의 입력 매핑이 단순해지지만, 그 단순함이 **장애물 패턴 디자인**의 복잡도로 전이된다. 10종 장애물 × 구역별 도입 순서 × 프로시저럴 변형의 조합 공간을 관리하지 않으면 후반 스테이지에서 "불공정한 즉사"가 빈번해질 수 있다. **장애물 도입 테이블(§7.2)을 구역별로 명확히 제한하고, 세그먼트 난이도 등급(E/M/H/X) 배분 공식을 기획서에 명시**해야 한다.
- **[Cycle 38]** 이전 사이클(#37)이 PENDING 상태로 전체 리뷰 미완료인 경우, 포스트모템에서 "아쉬운 점"과 "다음 사이클 제안"이 불완전할 수 있다. **PENDING 사이클의 기획서 설계 원칙(TRANSITION_TABLE, BFS 검증, 수학적 난이도 공식)은 계승하되, 구현 레벨 교훈은 검증 미완으로 표시**하여 과신하지 않아야 한다.
- **[Cycle 38]** 듀얼 시스템(퍼즐+건설, 경영+전투) 과대 문제가 Cycle 34~37에서 반복되었다. 본 사이클에서 **단일 핵심 루프(뒤집기→돌파→탈출)에 집중하고 서브시스템을 업그레이드 트리만으로 제한**하는 설계를 채택. Cycle 35 planner 교훈("시스템 수가 아닌 환경 변화 축의 복합성으로 깊이를 확보")을 직접 적용한 사례.
- **[Cycle 38]** 상태 머신을 4개(TITLE/MAP/PLAY/BOSS)로 더 단순화. Cycle 37의 5상태(+SHOP)에서 MAP에 상점 기능을 통합. **상태 수를 줄이는 것이 TRANSITION_TABLE 단순화와 직결되어 STATE_PRIORITY 재발 위험을 구조적으로 감소**시킨다. 단, GAMEOVER를 별도 상태가 아닌 PLAY 서브상태로 처리하는 설계가 핵심.
