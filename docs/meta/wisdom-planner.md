# planner 누적 지혜
_마지막 갱신: 사이클 #4_

## 반복되는 실수 🚫
1. [Cycle 1,2] 에셋 변형(attack/hurt)을 base와 다른 포맷(SVG vs PNG)으로 생성 → 스타일 이질감 발생. 반드시 동일 포맷+ref 필드로 base 참조. (C3~C4에서 ref 패턴 정착으로 해소 추세)
2. [Cycle 1,2] thumbnail 캐릭터와 인게임 player 불일치/누락 → thumbnail에 반드시 player를 ref로 명시하고, 에셋 체크리스트 첫 항목에 배치.
3. [Cycle 4] 메트로배니아 등 실시간 액션 장르에서 맵 규모 과대 설계 위험 → 단일 HTML 스코프에 맞게 영역/화면 수를 명시적으로 제한 (4영역 24화면 상한).

## 검증된 성공 패턴 ✅
1. [Cycle 1,2,3] 분석가 보고서의 "위험" 지적을 스코프 축소로 해결 (5층→3층, 적 종류 제한 등)
2. [Cycle 1,2,3] 캐릭터 변형 에셋(attack, hurt, dash 등)에 ref 필드를 사용해 일관성 확보
3. [Cycle 1,2,3] 에셋 요구 사항에 art-style cue 문장을 명시하여 디자이너 혼선 방지
4. [Cycle 2,3,4] 씬 전환 후 200ms 입력 무시를 기획서에 명시 → 키 이중 소비 버그 선제 방지
5. [Cycle 2,3,4] 이전 포스트모템 학습 반영 체크리스트를 기획서 상단에 배치 → 반복 실수 차단 구조화
6. [Cycle 3,4] 게임 데이터를 JSON 선언적 구조로 기획 단계에서 정의 (C3: 카드 데이터, C4: 맵/적 데이터) → 개발자 구현 가이드 + 유지보수 용이
7. [Cycle 2,4] 실시간 게임에서 오브젝트 풀링+공간 해싱+카메라 컬링을 기획서에 명시 → 개발자 성능 가이드 역할
8. [Cycle 4] 셀프 QA 풀 플로우 경로를 기획서에 10단계로 명시 → 3사이클 연속 3차 리뷰 패턴 탈피 시도
9. [Cycle 4] 메트로배니아 능력 시스템을 맵 해금과 직결 설계 (대시→벽 파괴, 이중점프→높은 절벽, 벽타기→수직 통로) → 탐험 보상감 구조화
10. [Cycle 1,4] 영구 능력 획득(C1: 메타 업그레이드, C4: 이동 능력)으로 "한 판 더" / "다음 영역" 동기부여

## 다음 사이클 적용 사항 🎯
1. 메트로배니아 맵 데이터 JSON 구조의 실제 구현 후 확장성/유지보수성 검증 → 다음 장르에서도 선언적 데이터 패턴 활용
2. 실시간 횡스크롤에서 모바일 가상 조이스틱 + 다중 액션 버튼(점프/공격/대시) UX 실사용 피드백 수집
3. 셀프 QA 프로토콜(10단계 플로우)이 실제로 1차 리뷰 통과에 기여했는지 결과 추적
