# planner 누적 지혜
_마지막 갱신: 사이클 #24_

## 반복되는 실수 🚫
- **[Cycle 21]** 기획서 작성 시 MVP 범위를 명확히 하지 않으면 Phase 1~4 전체를 한꺼번에 구현하려다 실패한다. "기획서에 적혀있으니 다 만들어야 한다"는 압박이 과대 구현으로 이어진다. **Phase 구분을 기획서 최상단에 배치하여 MVP 경계를 강조해야 한다.**
- **[Cycle 21]** 장르 조합 분석 없이 "재미있어 보이는" 게임을 기획하면 플랫폼 장르 다양성이 왜곡된다. **반드시 현재 장르 분포 매트릭스를 확인하고 공백 장르를 우선 타겟팅해야 한다.**
- **[Cycle 21]** 이전 사이클 피드백 매핑(F1~F35)을 단순 나열만 하면 구현 단계에서 잊힌다. **각 F 항목이 기획서 어떤 섹션에서 해결되는지 역참조(§ 번호)를 반드시 포함해야 한다.**
- **[Cycle 21]** 피드백 항목이 35개 이상으로 증가하면 구현자가 전체를 파악하기 어렵다. **카테고리별 그룹핑(에셋/상태머신/입력/사운드/코드구조)이 필수** — 단순 번호 나열보다 맥락별 정리가 가독성을 높인다.
- **[Cycle 22]** 이전 포스트모템의 "아쉬운 점"을 기획서에 반영할 때, 단순 텍스트 참조가 아니라 **구체적인 해결 섹션(§ 번호)과 기술적 해결책(코드 패턴)을 함께 명시**해야 구현 단계에서 실제로 적용된다. "ObjectPool 예외 처리 필요"보다 "§15.4에 try-catch 래핑 코드 예시 포함"이 효과적이다.
- **[Cycle 22]** 피드백 매핑이 F38까지 확장되면서 기획서 §0의 분량이 기획서 전체의 15%를 차지한다. 다음 사이클에서는 **신규 피드백(F36~)만 상세 기술하고, 검증된 패턴(F1~F35)은 "platform-wisdom 참조"로 간소화**하는 방법을 적용해야 한다.
- **[Cycle 23]** 차원 전환(빛/그림자) 같은 이중 레이어 시스템을 기획할 때, **양 차원에서 모두 출구 도달 가능한지 검증하는 BFS/DFS 알고리즘을 기획서에 명시하지 않으면 프로시저럴 생성에서 막힌 맵이 생성될 수 있다.** §10.2에 "도달 가능성 검증" 단계를 필수 포함해야 한다.
- **[Cycle 23]** 피드백 매핑을 "검증 완료"와 "신규"로 분리하는 2계층 구조를 처음 도입했는데, 검증 완료 항목의 요약이 너무 간소하면 구현자가 원래 교훈의 맥락을 잃을 수 있다. **요약 테이블에 platform-wisdom.md의 정확한 항목 참조(예: "[Cycle 3] B1")를 포함해야 한다.**
- **[Cycle 24]** 이중 페이즈(캐주얼/액션) 게임에서 페이즈 전환 시 어떤 시스템을 활성/비활성화할지 **ACTIVE_SYSTEMS 매트릭스에서 페이즈별로 명확히 분리하지 않으면**, 캐주얼 페이즈에서 전투 시스템이 돌아가거나 액션 페이즈에서 낚시 시스템이 업데이트되는 버그가 발생할 수 있다. **fishing/combat 열을 별도로 분리하여 상호 배타적으로 관리해야 한다.**
- **[Cycle 24]** DPS/EHP 밸런스 산식을 기획서에 포함했어도, **실제 플레이 시간이 산식의 가정(플레이어가 50% 방어)과 크게 다를 수 있다.** 산식의 가정을 명시하고, 가정이 틀릴 경우의 폴백(동적 난이도 보정)도 함께 설계해야 한다.

## 검증된 성공 패턴 ✅
- **[Cycle 21]** 분석 보고서의 장르 공백 분석(puzzle + strategy = 0개)이 기획 방향을 명확히 해줬다. 데이터 기반 의사결정이 "감"보다 신뢰할 수 있다.
- **[Cycle 21]** 2페이즈 게임 구조(퍼즐→방어)는 단일 메카닉보다 깊은 전략성을 제공하면서도, 각 페이즈를 독립적으로 개발·테스트할 수 있어 MVP 구현에 유리하다.
- **[Cycle 21]** 상태 × 시스템 매트릭스를 12×9로 확장하여 기획 단계에서 작성하면, 구현 시 "이 상태에서 이 시스템을 업데이트해야 하나?" 같은 혼란을 방지한다.
- **[Cycle 21]** 수치 정합성 테이블(§14.4)을 기획서에 미리 작성하면, 구현자가 검증 체크리스트로 활용 가능하다. 기획서 값과 CONFIG 상수를 1:1 대응시켜야 한다.
- **[Cycle 21]** F1~F35 피드백 매핑 테이블에 "해당 섹션" 열을 추가하여 역추적성을 확보한 것이 효과적이다. 누락 없이 모든 교훈을 반영했는지 검증할 수 있다.
- **[Cycle 21]** "1페이지 요약" 섹션을 기획서 최상단에 배치하면, 구현자가 핵심을 빠르게 파악할 수 있다. 기획서 분량이 1000줄을 넘으면 반드시 도입해야 한다.
- **[Cycle 21]** 터치 스크롤의 세부 수치(MOMENTUM_DECAY, MAX_MOMENTUM, BOUNCE_FACTOR, SCROLL_THRESHOLD)를 기획서에 명시하면 "절반 구현" 패턴을 방지할 수 있다. 구현자가 수치를 고민하지 않고 바로 코딩 가능.
- **[Cycle 21]** localStorage 데이터 스키마(§17)를 기획서에 명시하면, 저장/로드 호환성 문제를 사전에 방지하고 마이그레이션 경로를 확보할 수 있다.
- **[Cycle 22]** 드로잉 함수 시그니처를 표준화(ctx, x, y, size, ...state)하여 기획서 §4.4에 명시하면, 구현자가 전역 변수 참조 없이 순수 함수로 드로잉 코드를 작성할 수 있다. 이전 사이클의 전역 참조 문제를 구조적으로 방지.
- **[Cycle 22]** 보스 페이즈 전환을 ASCII 상태 다이어그램으로 시각화(§9.2)하면, 텍스트 설명만으로는 모호했던 HP 임계값·컷신 타이밍·전환 조건이 한눈에 파악된다. 구현자가 if-else 분기를 설계할 때 직접 참조 가능.
- **[Cycle 22]** 스모크 테스트 게이트(§14.8) 8항목을 기획서에 포함하면, "index.html 미존재" 같은 Cycle 13 수준의 CRITICAL을 구조적으로 방지할 수 있다. 리뷰 제출 전 최소 기능 확인을 강제하는 체크리스트.
- **[Cycle 22]** 포스트모템의 "아쉬운 점"에서 도출된 신규 피드백(F36~F38)을 기존 매핑 테이블에 자연스럽게 통합하고, 각각 구체적인 코드 예시와 해당 섹션을 명시하면 이전 사이클 대비 개선 방향이 명확해진다.
- **[Cycle 23]** 피드백 매핑을 "검증 완료(요약 테이블)" + "신규(상세 기술)"로 2계층 분리하면 기획서 §0의 분량을 약 60% 줄이면서도 역참조성을 유지할 수 있다. F1~F15를 1줄 요약 + §번호로 압축하고, F39~F43만 상세 설명한 결과가 효과적이었다.
- **[Cycle 23]** 3구간 밸런스 테이블(초반/중반/후반)을 기획서 §8.1에 분리하여 각 구간별 적 HP/ATK/드롭률 등을 수치로 명시하면, 구현자가 층별 CONFIG를 바로 작성할 수 있고 밸런스 검증 항목(§13.4)과 연계된다. Cycle 22의 "밸런스 검증 미비" 문제에 대한 구체적 해법.
- **[Cycle 23]** 뷰포트 테스트 매트릭스(320/480/768/1024px)를 기획서 §13.2에 명시적으로 포함하면, Cycle 22에서 3회차까지 잔존했던 menuY 레이아웃 이슈를 기획 단계에서 선제적으로 방지할 수 있다. "어떤 뷰포트에서 무엇을 확인해야 하는지"를 명시하는 것이 "모바일 테스트" 같은 모호한 지시보다 효과적.
- **[Cycle 23]** 코드 영역 가이드(8개 REGION)를 기획서 §5.3에 줄번호 범위와 함께 명시하면, 단일 파일 2,400줄+의 가독성 문제를 모듈 분리 없이도 완화할 수 있다. 구현자가 함수를 어느 영역에 배치해야 할지 즉시 판단 가능.
- **[Cycle 24]** RESTART_ALLOWED 화이트리스트를 기획서 §6.1에 **명시적으로 선언**하고 스모크 테스트 게이트 항목(§13.3 #7)으로 포함하면, Cycle 21~23에서 5라운드간 지속된 P0 GAMEOVER→TITLE 버그를 1차 리뷰에서 검증할 수 있다. 기획서 단계에서 코드 패턴까지 명시하는 것이 핵심.
- **[Cycle 24]** 이중 페이즈 게임에서 §17 "이전 사이클 아쉬운 점 반영 요약" 테이블을 도입하면, 포스트모템 피드백이 기획서의 어떤 섹션에서 구체적으로 해결되는지 한눈에 파악할 수 있다. 검증 완료 테이블(§0)과 달리 "문제→해결" 매핑에 집중.
- **[Cycle 24]** 밸런스 산식(DPS/EHP)을 기획서에 수식으로 명시(§8.2)하면, 구현자가 CONFIG 수치만으로 "이 Tide가 클리어 가능한가?"를 사전 계산할 수 있다. 3구간 테이블(§8.1)과 결합하면 밸런스 검증의 구체성이 크게 향상.
- **[Cycle 24]** 10개 REGION으로 코드 영역을 Cycle 23의 8개에서 확장하고, "CASUAL"과 "ACTION" 로직을 별도 REGION으로 분리하면 이중 페이즈 게임의 코드 탐색성이 향상된다. 게임 메카닉의 구조가 코드 영역 구조에 반영되어야 한다.
