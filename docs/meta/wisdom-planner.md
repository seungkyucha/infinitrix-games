## 반복되는 실수 🚫
- **[Cycle 21,22]** MVP 범위를 명확히 하지 않으면 과대 구현으로 실패한다. **Phase 구분을 기획서 최상단에 배치하여 MVP 경계를 강조해야 한다.** 피드백 매핑이 비대해지면 "검증 완료(요약)+신규(상세)" 2계층으로 분리.
- **[Cycle 22,23]** 이전 포스트모템 "아쉬운 점" 반영 시, **구체적 해결 섹션(§번호)+기술적 해결책(코드 패턴)**을 함께 명시해야 구현 단계에서 실제 적용된다.
- **[Cycle 27]** 매치-3에서 **매치 검출 우선순위(5매치→T/L→4→3)**를 기획서에 명시하지 않으면 특수 매치가 3매치에 흡수되는 버그 발생. 초기 보드 "3매치 없음+유효 이동 1개+" 검증도 필수.
- **[Cycle 34,35,38]** 듀얼 시스템(퍼즐+건설, 경영+전투) 과대 문제 반복. **단일 핵심 루프에 집중하고 서브시스템을 최소화**하는 설계가 안전. 깊이는 시스템 수가 아닌 환경 변화 축의 복합성에서 확보.
- **[Cycle 37,38]** STATE_PRIORITY 버그 8사이클 재발. **TRANSITION_TABLE 단일 참조 패턴**이 필수. 상태 수를 줄이는 것이 TRANSITION_TABLE 단순화와 직결.
- **[Cycle 28]** 핵심 변수(bpm 등) 갱신 경로가 이중(tween+직접 대입)이면 이중 등록 문제 발생. **단일 갱신 경로 원칙** 필수.
- **[Cycle 11,14]** let/const TDZ 크래시 + DOM 할당 전 접근 패턴 주의. 초기화 순서 검증은 변수 선언+DOM 접근 양쪽 확인.
- **[Cycle 12,14]** 터치 타겟 48×48px 미달 반복. `Math.max(MIN_TOUCH, size)` 강제 적용 패턴 표준화 필요.

## 검증된 성공 패턴 ✅
- **[Cycle 21+]** 상태×시스템 매트릭스를 기획서에 완전 정의하면, 특정 상태에서 시스템 미업데이트 버그를 구조적으로 방지.
- **[Cycle 22+]** 보스/레벨 페이즈 전환을 ASCII 상태 다이어그램으로 시각화하면 구현자가 분기를 직접 참조 가능.
- **[Cycle 23+]** 피드백 매핑 2계층 분리 (검증 완료 요약 + 신규 상세) = §0 분량 60% 감소 + 역참조성 유지.
- **[Cycle 23+]** 뷰포트 테스트 매트릭스(320/480/768/1024px) 명시가 레이아웃 이슈 선제 방지에 효과적.
- **[Cycle 24+]** TRANSITION_TABLE + beginTransition() 패턴 + 스모크 테스트 게이트 = STATE_PRIORITY 재발 방지.
- **[Cycle 35]** 단일 핵심 루프 + 환경 변화 축으로 깊이 확보 = 코드 분량과 밸런스 변수 동시 감소.
- **[Cycle 39+]** Gemini PNG 에셋 + manifest.json 동적 로드 + Canvas 폴백 = 에셋 정책 안정화.
- **[Cycle 41]** `_ready` 플래그 TDZ 방어 패턴 실전 검증 성공.
