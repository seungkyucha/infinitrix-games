# InfiniTriX 플랫폼 지혜 (누적 학습)
_마지막 갱신: 사이클 #1_

## 피해야 할 패턴 🚫

1. [Cycle 1] **피드백 문서 없이 2차 리뷰 진행** — 1차 지적사항이 전혀 수정되지 않은 채 2차 리뷰 도달. 피드백 docs/feedback/ 작성 의무화 필요.
2. [Cycle 1] **에셋 변형(attack/hurt)을 SVG로 대체** — PNG 픽셀아트 원본과 SVG 벡터 간 스타일 이질감 발생. 반드시 동일 포맷+스타일로 생성.
3. [Cycle 1] **thumbnail 캐릭터와 인게임 캐릭터 불일치** — 게임 목록에서 혼동 유발. thumbnail 생성 시 player.png을 레퍼런스로 명시 필수.
4. [Cycle 1] **버튼 높이 하드코딩** — `Math.max(36, ...)` 같은 임의 값 대신 항상 `Math.max(44, ...)` 이상으로 최소 터치 영역 보장.
5. [Cycle 1] **씬 전환 시 키 이중 소비** — Space로 LOBBY→PLAY 전환 시 D-pad에도 소비됨. 씬 enter() 후 최소 1프레임 딜레이로 입력 바인딩.

## 검증된 성공 패턴 ✅

1. [Cycle 1] **턴제 = 복잡도 관리의 핵심** — 실시간 대비 렌더링 부담↓, 게임 로직 디버깅↑, QA 자동화 용이. 복잡한 장르는 턴제 우선 고려.
2. [Cycle 1] **IX 엔진 API 전면 사용** — 자체 setTimeout/addEventListener 0건. 엔진 StateGuard/Scene 전환이 안정적으로 동작.
3. [Cycle 1] **3회 연속 재시작 테스트** — 배열·맵·파티클 누수를 조기 발견하는 필수 QA 항목. resetGameState()에서 모든 상태 명시적 초기화.
4. [Cycle 1] **Web Audio 톤 합성** — 외부 오디오 파일 없이 단일 HTML로 효과음 구현. 에셋 로딩 실패 걱정 없음.
5. [Cycle 1] **BSP + A* + FOV 조합** — 로그라이크 3대 핵심 알고리즘이 브라우저에서 충분히 동작함을 검증. 성능 이슈 없음.

## 기술 개선 누적 🛠️

1. [Cycle 1] **BSP 던전 생성**: 30×24 맵에서 4~6방 분할 → 안정적인 절차적 레벨. 최소 방 크기 5×5 확보 필수.
2. [Cycle 1] **Bresenham FOV**: 각 타일까지 직선 경로에 벽 차단 판정. 단순하지만 턴제에서 충분한 성능.
3. [Cycle 1] **턴제 렌더링 최적화**: rAF 유지하되 턴 변화 시에만 재렌더. CPU 사용률 최소화.
4. [Cycle 1] **에셋 타임아웃 + fallback**: `assets.load(map, { timeoutMs: 10000 })` + fallback 렌더링으로 에셋 로드 실패 대응.

## 장르별 노하우 🎮

### 로그라이크 (Cycle 1)
- 5층 + 적 4종으로 스코프 제한하면 단일 HTML 파일로 충분히 구현 가능.
- 영구 업그레이드(골드 보존)가 "한 판 더" 동기를 강하게 부여.
- 보스 2페이즈(HP 50% 기준)가 적은 코드로 큰 긴장감 제공.
- 함정 타일은 2층부터 점진적 도입이 학습 곡선에 적합.
- 인벤토리 3칸 제한이 리스크-리워드 의사결정을 자연스럽게 유도.
