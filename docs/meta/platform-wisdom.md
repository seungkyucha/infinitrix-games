# InfiniTriX 플랫폼 지혜 (누적 학습)
_마지막 갱신: 사이클 #12 (neon-drift-racer APPROVED, 2회차)_

## 피해야 할 패턴 🚫
- **[Cycle 1]** 범용 템플릿에서 에셋을 복사할 때 사용하지 않는 에셋을 남겨두면 불필요한 네트워크 요청이 발생한다 (player.svg, enemy.svg 등 3개 잔존)
- **[Cycle 1]** "외부 에셋 0개" 원칙을 세웠으면서 Google Fonts를 로드하는 등 기획 원칙과 구현의 불일치. 원칙을 세웠으면 코드 리뷰 단계에서 체크리스트로 검증해야 한다
- **[Cycle 1]** iframe 환경에서 `confirm()`/`alert()` 사용 불가 — 게임 내 확인 UI는 반드시 Canvas 기반 모달로 구현할 것
- **[Cycle 1]** 애니메이션을 "나중에 추가"하려고 setTimeout 잠금만 걸어두면, 결국 미구현으로 남는다. 이동 애니메이션은 핵심 피드백이므로 초기부터 구현해야 한다
- **[Cycle 2]** 기획서에 입력 모드별 동작 분기를 명시했어도, 코드에서 조건 변수를 선언만 하고 실제 분기에 사용하지 않으면 의도와 다르게 동작한다. 구현 후 기획서 대조 체크리스트 필수
- **[Cycle 2]** TweenManager 같은 핵심 유틸리티에서 `cancelAll()`과 `update()`가 동시에 호출될 수 있는 경로를 설계 시점에 차단해야 한다. 런타임 중 컬렉션 변경은 deferred 패턴으로 보호할 것
- **[Cycle 2]** **게임 상태 머신의 모든 상태에서 어떤 시스템을 업데이트해야 하는지 매트릭스를 만들지 않으면, tween·물리 등 핵심 시스템이 특정 상태에서 멈추는 버그가 생긴다.** CONFIRM_MODAL에서 tw.update()를 호출하지 않아 모달이 alpha=0으로 고정된 B1이 대표 사례. 상태 × 시스템 매트릭스를 기획서에 필수 포함할 것
- **[Cycle 2]** 저장(saveBest) 후에 비교(getBest)하면 항상 false — 상태 변경과 판정의 순서를 반드시 "판정 먼저, 저장 나중에"로 유지할 것 (B4)
- **[Cycle 2]** setTimeout으로 게임 상태 전환을 지연시키는 패턴은 Cycle 1에서 이미 문제가 되었는데 Cycle 2에서도 재발했다(B3). tween onComplete 콜백으로 완전 대체하는 규칙을 코딩 가이드라인에 명시할 것
- **[Cycle 3]** tween 기반 지연 전환(onComplete로 상태 변경) 시, 지연 구간 동안 원래 조건이 매 프레임 참으로 평가되어 콜백이 반복 호출될 수 있다. **반드시 가드 플래그(`waveClearing = true`)를 설정하여 1회만 실행되도록 보호할 것.** (B1: waveComplete 반복 호출로 골드/점수 무한 누적)
- **[Cycle 3]** 게임 내 복수의 상태 전환 tween이 동시에 활성화되면 나중에 완료되는 tween이 먼저 완료된 전환을 덮어쓴다. **상태 전환에 우선순위 개념이 필요** — 예: GAMEOVER 전환은 WAVE_PREP 전환보다 항상 우선. `if (lives <= 0) return;` 같은 사전 체크를 전환 함수 진입부에 배치할 것. (B2)
- **[Cycle 3]** 기획서에 "SVG 필터 미사용"을 명시해도 에셋 생성 단계에서 feGaussianBlur가 재발한다(Cycle 2 B5 → Cycle 3 B4). 기획서 원칙은 코드뿐 아니라 에셋 파일에도 자동 검증 스크립트를 적용해야 실효성이 있다.
- **[Cycle 3]** 변수를 선언만 하고 실제로 갱신/사용하지 않는 "유령 코드"가 반복된다(Cycle 2: 입력 모드 분기, Cycle 3: livesAtWaveStart). **선언된 변수가 기획서 의도대로 사용되는지 코드 리뷰 체크리스트에 포함할 것.**
- **[Cycle 4]** TweenManager의 deferred `cancelAll()` 직후 `add()`를 호출하면 `_pendingCancel` 플래그가 신규 tween까지 삭제한다. **cancelAll과 add 사이의 경쟁 조건은 deferred 패턴의 맹점이다.** `clearImmediate()` 같은 즉시 정리 API를 분리하거나, cancelAll 직후 `_pendingCancel = false` + `_tweens.length = 0`으로 플러시해야 한다. (B1: 게임 시작 불가 CRITICAL)
- **[Cycle 4]** SVG 에셋 재발이 3사이클 연속(Cycle 2→3→4)이다. "기획서에 금지 명시"만으로는 에셋 생성 단계의 위반을 막을 수 없음이 확정적으로 증명됨. **자동 grep 검증 스크립트 없이는 해결 불가.**
- **[Cycle 5]** 자동 검증 스크립트를 도입해도, 코드 완성 후에만 돌리면 초기 코드에 잔존 패턴(ASSET_MAP, SPRITES, preloadAssets)이 1회차 리뷰에서 발견된다. **검증은 코딩 중에도 수시로 실행해야 효과적이다.** 코드 완성 후 1회 실행으로는 부족.
- **[Cycle 5]** 단일 변수(bpm)에 대해 tween 갱신과 직접 대입이 동시에 존재하면 이중 등록 문제가 발생한다. **하나의 값에 대한 갱신 경로는 반드시 하나로 통일할 것.** (1회차 B2: bpmObj tween + bpm 직접 반영 중복)
- **[Cycle 5]** 범용 템플릿에서 assets/ 디렉토리를 복사하면 코드에서 미참조하는 SVG 파일이 계속 잔존한다. 코드 레벨 검증만으로는 파일 잔존을 막을 수 없으므로, **에셋 디렉토리 자동 정리 또는 미참조 에셋 경고 스크립트가 필요하다.**
- **[Cycle 5]** 기획서에서 "모든 상태 전환에 beginTransition() 필수"를 명시해도 startGame()/goToTitle() 같은 직접 전환이 발생한다. **기획서 원칙과 구현의 불일치는 코드 리뷰 체크리스트에서 명시적으로 검증해야 한다.** 기능적 문제가 없어도 전환 애니메이션 누락으로 UX가 저하된다.
- **[Cycle 6]** 3단계 자동 검증(코딩 전/중/후)을 기획했음에도 1회차 리뷰에서 assets/ 디렉토리가 잔존 발견. **검증 스크립트는 기획서에 명시하는 것만으로는 부족하고, CI/빌드 파이프라인에 강제 실행되도록 통합해야 실효성이 있다.**
- **[Cycle 6]** 전역 객체에 직접 의존하는 유틸 함수(`speed()`가 `ball`에 직접 접근)는 테스트 불가. **게임 로직 함수는 파라미터를 통해 데이터를 받는 순수 함수 패턴으로 작성해야 단위 테스트가 가능하다.**
- **[Cycle 7]** assets/ 디렉토리 + SVG 에셋 재발이 7사이클 중 대부분에서 반복. Cycle 5~6에서 "종결" 선언 후에도 1회차 리뷰에서 재발 발견(2회차에서 삭제 완료). **기획서 명시·자동 검증 스크립트·3단계 체크리스트 모두 실효성이 제한적**임을 확정. CI/빌드 훅에서 `assets/` 존재 시 빌드 실패하는 강제 장치만이 근본 해결책이다.
- **[Cycle 7]** 순수 함수 설계 원칙을 기획서에 명시하고 대부분 준수했으나, 1회차 리뷰에서 전투/충돌 함수 5개의 전역 P 직접 참조가 발견(2회차에서 모두 파라미터화 수정). **원칙을 세우면 전수 검증 체크리스트로 "모든" 함수를 확인해야 한다.**
- **[Cycle 7]** 기획서 수치(크리티컬 Lv3=30%)와 구현 수치(25%)의 불일치. **기획-구현 정합성 검증이 기능 존재 여부뿐 아니라 수치 레벨까지 내려가야 한다.**
- **[Cycle 8]** assets/ 디렉토리 재발이 8사이클 연속. 기획서 §13.1에 pre-commit 훅 스크립트까지 명세했으나 실제 파이프라인에 등록하지 않아 무효. **기획서에 "스크립트를 작성하라"고 쓰는 것과 실제로 `.git/hooks/`에 등록하는 것은 완전히 다른 단계다.** 다음 사이클에서는 훅 등록 자체를 검증 항목에 포함해야 한다.
- **[Cycle 8]** `beginTransition()` 경유 원칙을 세워도 PAUSED처럼 "즉각 전환이 자연스러운" 상태에서 직접 `enterState()` 호출이 발생한다. **원칙의 예외를 허용할 것인지, 아니면 beginTransition()에 즉시 전환 모드를 추가할 것인지** 설계 결정이 필요하다.
- **[Cycle 10]** 수정 과정에서 새 버그를 도입하는 "수정 회귀" 패턴 발생. 1회차 MINOR 4건 수정 중 `render()` 시그니처 변경 시 `timestamp` 미전달로 CRITICAL 신규 도입(2회차). **수정 후 전체 플로우 회귀 테스트(TITLE→MAP→BATTLE→VICTORY/DEFEAT)를 반드시 수행할 것.** 개별 수정 검증만으로는 부족하다.
- **[Cycle 10]** 기획서 game-id(`mini-card-battler`)와 실제 배포 디렉토리명(`mini-card-crawler`)이 불일치. 기획→구현 과정에서 네이밍 변경 시 기획서도 갱신해야 추적 가능성이 유지된다.
- **[Cycle 10]** 코드에서 에셋 참조가 0건이어도 assets/ 디렉토리가 물리적으로 존재하면 기획 위반. **코드 레벨 클린 ≠ 파일 시스템 레벨 클린.** `fetch`, `Image()`, `assets` 문자열 검색만으로는 불필요 파일 잔존을 감지할 수 없다. pre-commit 훅에서 디렉토리 존재 자체를 차단해야 한다.
- **[Cycle 10]** 극단적 축약 변수명(`shopI` vs `shI`)의 불일치가 CRITICAL 크래시(상점 진입 시 ReferenceError)를 유발. **코드 압축을 위한 축약 시 함수 내 변수 선언과 참조의 일관성을 반드시 확인할 것.** 'use strict' 모드가 미선언 변수를 런타임에 잡아주지만, 선언 자체가 다른 이름으로 되어 있으면 방지 불가. 축약 네이밍 규칙(접두사 패턴 등)을 사전에 정의하는 것이 안전하다.
- **[Cycle 11]** `let gridCache` 선언이 `resizeCanvas()` 호출보다 뒤에 위치하여 TDZ(Temporal Dead Zone) 크래시 발생 → 게임 완전 불능. **`let`/`const` 변수는 반드시 최초 사용(함수 호출 포함) 이전에 선언되어야 한다.** 변수 선언 순서라는 기본적인 실수가 CRITICAL로 이어질 수 있으므로, 초기화 코드 블록의 실행 순서를 명시적으로 검증할 것.
- **[Cycle 11]** 아이들 게임에서 탭 전환(프레스티지/통계) 시 `updateProduction()`을 호출하지 않으면 생산이 멈추는 문제. **아이들/방치형 장르에서는 UI 상태와 무관하게 생산 시스템이 항상 동작해야 한다.** 상태×시스템 매트릭스에 "백그라운드 시스템"을 별도 분류할 것.
- **[Cycle 11 mini-platformer]** assets/ 디렉토리 + SVG 에셋이 11사이클 연속 재발. 기획서 §12.1에서 "절대 금지"로 특별 강조 + platform-wisdom에서 10사이클 연속 재발 경고를 했음에도 10개 SVG 파일 생성. **기획서 명시, 자동 검증 스크립트, 체크리스트, 피드백 매핑 모두 실효성 한계 확정. CI/빌드 파이프라인에서 디렉토리 존재 시 빌드 실패하는 강제 장치만이 근본 해결책.** → 3회차 리뷰에서 완전 삭제 확인, APPROVED.
- **[Cycle 11 mini-platformer]** 3회 리뷰 사이클 소요: 1회차 CRITICAL(assets/ 10 SVG) → 2회차 CRITICAL(ASSET_MAP/SPRITES/preloadAssets 코드 잔존) → 3회차 APPROVED. **에셋 삭제 후에도 관련 코드(ASSET_MAP, SPRITES, preloadAssets, if(SPRITES.xxx) 분기)가 잔존하면 CRITICAL로 재판정된다.** "파일 삭제"와 "코드 정리"를 반드시 동시에 수행할 것.
- **[Cycle 11 mini-platformer]** gameLoop try-catch 래핑이 기획서 §12.9에 명시되었음에도 미적용. 이전 사이클(Cycle 10)에서 동일 패턴을 수정했으나 다음 게임에서 재발. **방어적 코딩 패턴도 게임 간 전파가 자동화되지 않으면 매번 누락 위험.** 공용 엔진 모듈 분리가 근본 해결.
- **[Cycle 11 mini-platformer]** 기획서 명시 기능(일일 챌린지 §6.5) 완전 미구현 + 미사용 스프라이트 3건 로드. **기능 구현 체크리스트에서 "존재 여부"뿐 아니라 "미사용 코드/에셋" 역방향 검증도 필요.**
- **[Cycle 12]** assets/ 디렉토리 12사이클 연속 재발. 이번에는 코드 내 에셋 참조 0건(순수 Canvas 드로잉 달성)이라는 진전이 있으나, 파일 자체(manifest.json + SVG 8개)는 여전히 물리적으로 존재. **"코드 레벨 클린"과 "파일 시스템 레벨 클린"은 별개 문제라는 Cycle 10 교훈이 재확인됨.** CI pre-commit 훅의 실제 등록이 더 이상 미룰 수 없는 과제.
- **[Cycle 12]** 기획서 §5.3 물웅덩이 조향 반전 — 1차 리뷰에서 puddleTimer와 속도 감소만 구현되고 조향 반전이 누락된 것이 발견됨. 최종 리뷰에서 `steerMul = -1` 로직 추가로 해결(L545). **"절반 구현"은 코드 리뷰에서 발견하기 어려운 패턴이다.** 기획서의 "A + B" 명세에서 A만 구현되고 B가 빠지는 경우, 기능별 세부 체크리스트(A: ✅, B: ☐)가 필요. → **최종 리뷰에서 해결됨.**
- **[Cycle 12]** 트랙 선택 화면에서 매 프레임 `loadTrackData()` 호출(5트랙 × 60fps = 초당 300회 localStorage 읽기). **화면 진입 시 1회 캐싱이면 해결되는 패턴이지만, "동작은 하니까" 놓치기 쉽다.** 성능 관련 코드 리뷰 항목에 "반복 호출 내 I/O 접근" 체크를 추가할 것.
- **[Cycle 12 2회차]** 일시정지 버튼 40×40px → WCAG 터치 타겟 최소 44×44px 미달. **터치 타겟 최소 크기(44×44px)를 기획 단계에서 UI 스펙에 명시적으로 포함할 것.** 구현 후 리뷰에서 잡는 것보다 기획서 수치 테이블에 포함하는 것이 효율적.
- **[Cycle 12 2회차]** 빈 if 블록(데드 코드)이 코딩 중 임시로 남겨져 1회차에서 지적됨. **임시 코드 블록에는 TODO 주석을 달거나, 작업 완료 시 즉시 정리하는 습관이 필요.** 빈 블록은 "의도적 미구현"인지 "실수"인지 리뷰어가 판단할 수 없다.

## 검증된 성공 패턴 ✅
- **[Cycle 1]** 단일 HTML 파일 + Canvas API + Vanilla JS 조합은 로딩 속도와 iframe 호환성 모두 우수하다. 첫 게임 아키텍처로 검증 완료
- **[Cycle 1]** 게임 상태 머신 (LOADING→TITLE→PLAYING→GAMEOVER) 패턴은 상태 전환 버그를 원천 차단한다. 모든 게임에 적용할 것
- **[Cycle 1]** `Promise.all` + `img.onerror = resolve` 에셋 프리로드 + 코드 폴백 렌더링 전략은 에셋 실패에도 게임이 동작하는 견고한 패턴
- **[Cycle 1]** DPR 대응 (Canvas 내부 해상도 ≠ CSS 표시 크기)은 고해상도 디바이스에서 시각 품질을 크게 올려준다
- **[Cycle 1]** localStorage를 try-catch로 감싸면 iframe sandbox 환경에서도 안전하게 동작한다
- **[Cycle 1]** 기획서에 HEX 코드, 점수 테이블, 수식까지 구체적으로 명시하면 구현 충실도가 92%까지 올라간다
- **[Cycle 2]** TweenManager(lerp + easing)를 게임 초기부터 내장하면 적 등장/폭발/UI 전환 등 16곳 이상에서 재사용되어 코드 중복이 사라진다. 모든 시각적 전환은 tween으로 통일할 것
- **[Cycle 2]** ObjectPool(acquire/release)은 슈팅 게임의 총알·적·파티클 같은 고빈도 생성/소멸 엔티티에서 GC 스파이크를 제거한다. 역순 순회 + splice 패턴은 안전한 release를 보장
- **[Cycle 2]** 게임 상태 머신을 6상태(LOADING/TITLE/PLAYING/PAUSE/CONFIRM_MODAL/GAMEOVER)로 확장해도 구조가 깔끔하게 유지된다. Canvas 모달은 confirm() 완벽 대체 (단, 모든 상태에서 tween 업데이트 필수)
- **[Cycle 2]** 이전 사이클의 문제점을 기획서 §0에 명시적 테이블로 정리하고, 각각의 해결 방법을 사전에 설계하면 같은 실수가 반복되지 않는다. "피드백 반영 섹션"을 기획서 표준으로 채택할 것
- **[Cycle 2]** 보스 AI의 3페이즈 순환 패턴(상태 머신 위의 상태 머신)은 적은 코드로 복잡한 행동을 만든다. HP 기반 난이도 스위칭도 효과적
- **[Cycle 2]** 플레이어 히트박스를 시각 크기(40×48)보다 작게(24×24) 설정하는 "관대한 판정"은 슈팅 게임에서 체감 난이도를 적절히 낮춰준다. 앞으로 액션 게임에서 표준 적용할 것
- **[Cycle 3]** 상태×시스템 업데이트 매트릭스를 기획서에 정의하고 코드 주석으로 이중 포함하는 패턴이 실제로 효과적임을 검증. Cycle 2의 CONFIRM_MODAL tween 버그(B1/B2)가 Cycle 3에서 완전히 재발하지 않았다. **매트릭스는 기획서 필수 항목으로 확정.**
- **[Cycle 3]** 이전 사이클 미해결 항목(destroy() 패턴, easeOutElastic, setTimeout 금지)을 기획서 §0에서 명시적으로 추적하고 해결하는 워크플로우가 3사이클 연속 동작. registeredListeners + listen() 헬퍼 + destroy()로 이벤트 리스너 lifecycle 문제 해결 완료.
- **[Cycle 3]** Web Audio API 절차적 사운드는 외부 에셋 0개 원칙을 유지하면서 게임에 청각 피드백을 추가하는 실용적 방법. OscillatorNode + GainNode + try-catch 래핑으로 미지원 환경에서도 안전.
- **[Cycle 3]** 타워 디펜스에서 First 타겟팅(웨이포인트 인덱스 + 구간 진행률)은 구현이 간단하면서도 플레이어에게 직관적으로 느껴지는 타겟팅 방식이다. 관대한 히트박스(+8px)도 전략 장르에서 유효.
- **[Cycle 4]** TransitionGuard 패턴(`STATE_PRIORITY` 맵 + `beginTransition()` 헬퍼)은 Cycle 3 B1/B2의 근본 해결책으로 설계되었고, GAMEOVER 전환 우선순위 등 의도한 대로 동작함을 확인. **상태 전환 표준 패턴으로 확정.**
- **[Cycle 4]** 유령 변수 방지 체크리스트(기획서에 명시된 변수의 갱신/사용처 전수 검증)가 실제로 효과적. `inputMode` 4곳, `nearMissCount`, `timeSinceLastPowerup`, `consecutiveSafeDist` 모두 정상 동작 확인. Cycle 2~3의 반복 이슈 해결.
- **[Cycle 4]** 절차적 청크 생성(400px 단위, 6종 가중 패턴, 안전 레인 보장)은 무한 러너에서 "불가능한 배치" 없이 매번 다른 경험을 제공하는 검증된 접근. 안전 규칙(최소 1레인 통과 가능, 연속 레이저 금지)이 핵심.
- **[Cycle 4]** Near Miss 시스템(장애물 통과 시 y거리 판정 + 슬로모 tween + 점수 보상)은 구현 비용 대비 긴장감·보상감 효과가 뛰어나다. 액션/아케이드 장르에 범용 적용 가능.
- **[Cycle 5]** `enterState()` 패턴(상태 진입 시 초기화 일원화)은 Cycle 4 타이틀 글로우 미복구 문제를 구조적으로 해결. switch(state)로 각 상태 진입 시 필요한 tween/오디오 초기화를 한 곳에서 관리하면 누락이 불가능하다. **상태 진입 초기화 표준 패턴으로 확정.**
- **[Cycle 5]** `clearImmediate()` API 분리(cancelAll=deferred vs clearImmediate=즉시)가 Cycle 4 B1 CRITICAL을 완전히 해결. resetGame()/goToTitle()에서 clearImmediate()를 사용하여 cancelAll+add 경쟁 조건을 원천 차단. **TweenManager 최종 안정 버전으로 확정.**
- **[Cycle 5]** 자동 grep 검증 스크립트(§13.5)가 4사이클 연속 재발했던 SVG 에셋 문제를 최종 해결. 금지 패턴 7개 항목 자동 검증으로 코드 리뷰 전 PASS/FAIL을 기계적으로 판정. **모든 사이클에서 필수 실행 항목으로 확정.**
- **[Cycle 5]** Web Audio API를 효과음(C3~C4)에서 **핵심 게임플레이 메커니즘으로 격상** 성공. AudioContext.currentTime 기반 고정밀 비트 스케줄링(scheduleAhead 100ms + lookAhead 50ms)으로 드럼/베이스/멜로디 3트랙 실시간 합성 + 노트 판정 동기화 달성. setTimeout 완전 배제.
- **[Cycle 5]** 절차적 비트맵 생성(난이도별 레인 수/노트 유형/밀도 조절 + 안전 규칙 4종)은 리듬 게임에서 패턴 암기가 아닌 반응 테스트를 가능하게 한다. Cycle 4의 절차적 청크 생성과 동일한 설계 철학이 장르를 넘어 유효.
- **[Cycle 5]** offscreen canvas 캐시(격자 그리드) + globalAlpha 조절로 비트 반응형 배경을 구현하는 패턴은 매 프레임 복잡한 배경 재드로잉 없이 시각적 역동성을 달성하는 효율적 기법.
- **[Cycle 5]** 4×8 그리드 + 5종 음표 + 3매치 룰의 조합은 테트리스와 매치3의 장점을 취하면서 Web Audio 화음 생성과 자연스럽게 결합된다. MatchSystem의 O(32) 경량 매치 판정은 체인 재귀에서도 성능 부담이 없다.
- **[Cycle 5]** 1회차 코드 리뷰 APPROVED 달성 — 5사이클 기획서 피드백 반영 누적 워크플로우(§0 피드백 매핑 + §13.4 대조 체크리스트)가 초회 통과율을 높이는 데 실질적으로 기여함을 확인.
- **[Cycle 5]** 5종 블록을 색상+도형(원/다이아/삼각/사각/별)+글자로 3중 구분하는 접근은 색각이상 접근성을 확보하면서 시각적 다양성도 높인다. 접근성 설계는 추가 비용 없이 게임 품질을 올리는 패턴.
- **[Cycle 6]** assets/ 디렉토리를 아예 생성하지 않는 원칙 + 금지 패턴 자동 검증 3단계(코딩 전/중/후)의 조합으로 4사이클 연속 재발했던 SVG 에셋 문제를 최종 종결. **"생성 자체를 금지"가 "사후 정리"보다 확실하다.**
- **[Cycle 6]** 외부 물리 라이브러리 없이 Vector2 클래스 + 법선 반사 + 마찰 감속만으로 미니 골프 물리를 충분히 구현 가능. Canvas API 코드 드로잉과 기존 인프라(TweenManager, ObjectPool, TransitionGuard)가 물리 장르에서도 안정적으로 동작함을 검증.
- **[Cycle 6]** 궤적 프리뷰(1차 반사까지 20점 시뮬레이션)는 "예측 가능한 물리"에 대한 플레이어 신뢰를 주면서도 2차 반사 이후를 숨겨 도전 요소를 유지하는 균형 잡힌 설계. 물리 퍼즐 장르의 핵심 UX 패턴.
- **[Cycle 6]** 관대한 홀 판정(시각 16px < 판정 20px) + 속도 임계치(speed < 2.0)의 조합은 Cycle 2의 관대한 히트박스 철학을 물리 장르에 성공적으로 적용한 사례. "아슬아슬하게 들어간" 쾌감이 리플레이 동기를 높인다.
- **[Cycle 6]** 포탈 쿨다운(30프레임)으로 무한 루프 방지, 움직이는 벽의 sin 기반 왕복으로 예측 가능한 타이밍 퍼즐 구현 — 간단한 수학 함수로 복잡한 레벨 메커닉을 만드는 패턴이 유효.
- **[Cycle 6]** 기획서 충실도 22/22(100%) + 금지 패턴 0건 + 콘솔 에러 0건. 6사이클간 누적된 피드백 워크플로우(§0 피드백 매핑 + §13 체크리스트 + 자동 검증)가 성숙 단계에 도달했음을 확인.
- **[Cycle 7]** 순수 함수 설계 원칙(§10)을 기획서에 명시하고 `updatePlayer()`, `checkCircleCollision()`, `getWaveConfig()` 등 주요 함수를 파라미터 기반으로 구현 성공. Cycle 6의 전역 의존 문제를 구조적으로 개선한 첫 사례. **순수 함수 목록 + 파라미터 시그니처를 기획서에 미리 정의하는 패턴이 효과적.**
- **[Cycle 7]** 선언적 웨이브 config(`getWaveConfig(wave)`)로 20웨이브 난이도 곡선을 수학 공식 하나로 제어. 하드코딩 최소화 + 밸런스 조정이 상수 변경만으로 가능. Cycle 2~3의 공식 기반 스케일링 패턴이 서바이버 장르에서도 유효함을 재확인.
- **[Cycle 7]** 시드 기반 RNG(`seededRng()` LCG)로 일일 챌린지 구현 — 동일 날짜에 동일 적 스폰/스킬 풀 보장. 서버 없이 localStorage만으로 비동기 경쟁감을 추가하는 경량 패턴. **Cycle 6에서 제안된 비동기 경쟁 시스템의 첫 프로토타입으로 유효.**
- **[Cycle 7]** ObjectPool 4종(적100+투사체200+젬200+파티클300=800개) 동시 운용이 안정적. 역순 순회+splice 패턴이 매 프레임 수십 건 acquire/release에서도 GC 스파이크 없이 동작. 서바이버 장르의 대량 엔티티 관리에 ObjectPool이 필수임을 확인.

- **[Cycle 7]** 2회차 리뷰에서 순수 함수 18개 전수 검증 완료(전역 참조 0건, rng 참조는 설계 의도로 허용). 1회차 지적 → 2회차 수정이라는 피드백 루프가 코드 품질 보증에 효과적임을 확인. **기획서에 함수 시그니처를 미리 정의 + 리뷰에서 전수 검증하는 2단계 접근이 유효.**

- **[Cycle 8]** 순수 함수 18개 + 전역 참조 0건이 2사이클 연속(Cycle 7→8) 달성. "기획서에 함수 시그니처 사전 정의 → 리뷰 전수 검증"의 2단계 접근이 **장르와 무관하게 범용적으로 유효**함을 확인(액션/서바이버 + 경영/타이쿤).

## 장르별 노하우 🎮
- **퍼즐 [Cycle 1]**: 5×5 그리드 + 슬라이드 머지 메카닉은 구현 난이도 대비 재미 효율이 높다. 핵심은 "합치면 진화"의 시각적 보상. `merged[][]` 배열로 한 턴 중복 머지를 방지하는 패턴은 2048 계열 필수 기법. 동적 난이도(점수별 블록 분포 변화)는 명시적 난이도 선택 없이도 자연스러운 긴장감 상승을 만든다.
- **아케이드/슈팅 [Cycle 2]**: 종스크롤 슈팅의 핵심은 "즉각적 타격감"과 "점진적 파워 판타지". 스크린 셰이크 + 폭발 파티클 + 점수 팝업의 3종 세트가 한 발 명중의 쾌감을 만든다. 웨이브 시스템은 `spawnInterval = max(400, 1800 - wave × 60)`처럼 공식 기반 스케일링이 수동 테이블보다 유지보수가 쉽다. 플레이어 히트박스를 시각 크기(40×48)보다 작게(24×24) 설정하면 "운 나쁜 사망"이 줄어 체감 난이도가 적절해진다. 보스는 3페이즈 순환 + HP 기반 가속의 조합으로 적은 코드로 깊이 있는 전투를 만들 수 있다. **주의점:** 게임 상태가 6종 이상으로 늘어나면 각 상태에서 돌아야 할 시스템(tween, physics)을 빠뜨리기 쉽다 — 매트릭스 필수.
- **전략/타워디펜스 [Cycle 3]**: 타워 디펜스의 핵심 재미는 "제한된 자원으로 최적 배치를 찾는 퍼즐적 쾌감"과 "파워 성장의 판타지"이다. 3종 타워(빠른 단일/범위 감속/폭발 광역)의 상성 삼각형은 적은 종류로 깊은 전략을 만든다. S자 고정 경로 + First 타겟팅(웨이포인트 인덱스 + 진행률)은 구현이 간단하면서 직관적. 웨이브 스케일링 공식(`min(20, 5+N×1.5)` 적 수, `max(500, 2000-N×80)` 스폰 간격)은 수동 테이블보다 유지보수 용이. **주의점:** 타워 디펜스는 게임 루프 내 시스템(웨이브 매니저, 적 이동, 타워 공격, 투사체, 파티클, 경제)이 슈팅보다 많아 상태 전환 시 경쟁 조건이 발생하기 쉽다. tween 지연 전환에는 반드시 가드 플래그를 걸고, 전환 간 우선순위(GAMEOVER > WAVE_PREP)를 명시할 것. 동적 밸런스 보정(위기 보너스, 숙련자 도전)은 좋은 아이디어지만 실제 동작 검증을 테스트에 포함해야 한다.
- **캐주얼/무한러너 [Cycle 4]**: 무한 러너의 핵심은 "한 판만 더"의 중독성과 "즉각적 반응의 쾌감"이다. 3레인 + 점프라는 단순한 조작이 진입 장벽을 낮추고, 거리 비례 속도 증가(`min(600, 200 + distance × 0.04)`)가 자연스러운 난이도 곡선을 만든다. 절차적 청크 생성(6종 가중 패턴 + 안전 레인 보장)은 패턴 암기가 아닌 순간 판단력 테스트로 리플레이 가치를 높인다. Near Miss(아슬아슬 회피 +30점 + 슬로모)는 적은 구현 비용으로 긴장감·보상감을 크게 올리는 범용 메커니즘. **주의점:** TweenManager의 cancelAll 직후 add 시나리오를 반드시 테스트할 것 — deferred 패턴의 맹점이 게임 시작 불가 CRITICAL 버그를 유발했다. 또한 "100% Canvas 드로잉" 원칙을 세웠으면 에셋 생성 단계까지 자동 검증이 필요하다.
- **리듬퍼즐/캐주얼 [Cycle 5]**: 리듬 퍼즐의 핵심 재미는 "퍼즐을 풀면 음악이 만들어진다"는 인과 쾌감이다. 4×8 그리드 + 5종 음표(C메이저 스케일) + 3매치 룰의 조합은 테트리스의 배치 전략과 매치3의 정렬 쾌감을 자연스럽게 결합한다. Web Audio API의 `sfxClearChord()`로 매치된 음계의 화음을 동시 연주하고, 체인마다 옥타브 위 + 베이스를 겹쳐 풍성해지는 사운드 디자인이 시각+청각 동시 보상을 만든다. C메이저 스케일 내 음만 사용하면 불협화음이 배제되어 안전하다. 레벨별 음표 풀 확장(3→4→5종)은 적은 코드로 자연스러운 난이도 곡선을 만든다. 동적 밸런스 보정(위기 시 매치 유리 블록 +50%, 연속 미매치 시 강제 보정)으로 "불가능한 상황"을 방지한다. 5종 블록의 색상+도형+글자 3중 구분은 색각이상 접근성 확보의 좋은 사례. **주의점:** 4열 그리드에서 3매치 빈도를 정밀 밸런싱하려면 실플레이 데이터가 필요하다. headless 테스트로는 밸런스 체감을 검증할 수 없다.
- **물리퍼즐/캐주얼 [Cycle 6]**: 미니 골프의 핵심 재미는 "궤적 예측의 전략적 쾌감"과 "물리 피드백의 직관성"이다. 드래그 에이밍 + 점선 궤적(1차 반사까지)으로 "이 각도로 치면 들어간다"를 계획하는 당구적 사고가 발생하고, 법선 반사·마찰·에너지 손실이 현실 물리를 직관적으로 체감하게 한다. 10레벨 점진적 요소 도입(빈 공간→벽→모래→물→움직이는벽→포탈→복합)은 매 레벨 새 메커닉을 학습하는 완만한 곡선을 만든다. 별 3개 시스템(Par 기준)과 홀인원 보너스(+500)가 리플레이 동기를 강력히 부여한다. 포탈 쿨다운(30프레임)은 무한 루프 방지에 효과적이고, 움직이는 벽의 sin 왕복은 예측 가능한 타이밍 퍼즐로 작동한다. **주의점:** 물리 가속(Space 4배속) 시 서브스텝 없이 반복 루프로 충돌을 처리하면 고속에서 관통이 발생할 수 있다. 레벨 데이터 하드코딩은 확장성 제약 — 레벨이 20개 이상이면 JSON 분리 검토. 전역 상태(`ball` 직접 참조)를 파라미터화하면 테스트 가능성이 높아진다.
- **액션/서바이버 [Cycle 7]**: 뱀서라이크 서바이버의 핵심 재미는 "빌드 크래프팅(매 레벨업 스킬 선택 → 매 판 다른 조합)"과 "성장의 판타지(1발 → 화면 가득 탄막)"이다. 12종 스킬(공격 6/유틸 3/방어 3) 풀에서 3개 랜덤 제시 + 최대 레벨 제외로 자연스러운 빌드 다양성이 발생한다. 자동 공격은 "이동에만 집중"이라는 간결한 조작으로 진입 장벽을 낮추면서, 적 틈새를 찾아 빠져나가는 마이크로 플레이에서 깊이를 확보한다. `getWaveConfig(wave)` 공식 기반 스케일링으로 20웨이브 밸런스를 상수 조정만으로 제어. 콤보 시스템(2초 타임아웃 ×3.0 최대)은 공격적 플레이를 보상하는 간단하면서 효과적인 메커니즘. 시드 기반 일일 챌린지(`seededRng()` LCG)로 서버 없이 비동기 경쟁감 추가 가능. ObjectPool 4종(800개)이 대량 엔티티 동시 관리에 필수. **주의점:** 순수 함수 원칙을 세워도 일부 함수에서 위반이 발생하므로 전수 검증이 필요하다. 에셋 재발 문제는 기획서·스크립트 레벨 대응의 한계를 확정 — CI 강제가 유일한 해결책.
- **경영/타이쿤 [Cycle 8]**: 하이브리드 캐주얼 타이쿤의 핵심 재미는 "수동 제조의 쾌감(재료 순서 맞추기 → 팁 보너스)"과 "경영 성장의 판타지(노점→프랜차이즈)"이다. 고객 4타입(일반/급한/VIP/단골)의 인내심·팁 배율 차이가 "누구를 먼저 서빙할까"라는 마이크로 의사결정을 유발하고, 바리스타 AI 자동 제조와 플레이어 수동 제조의 공존이 "자동화해도 수동이 더 이득"이라는 설계를 유지한다. Day 기반 30일 진행(하루 30초)은 세션 길이를 15~20분으로 통제하면서, 매일 정산→업그레이드→다음 날의 루프가 "한 번만 더" 중독성을 만든다. 아이들 매장(2·3호점)의 수동 대비 낮은 수익률이 "플레이어 직접 운영의 가치"를 보존한다. CONFIG 객체 1:1 매핑 + §13.5 수치 검증 테이블로 기획-구현 정합성 100% 달성. **주의점:** 경영 게임은 시스템 수(고객/제조/바리스타/경제/만족도/매장확장)가 많아 상태×시스템 매트릭스가 필수. PAUSED 같은 "즉각 전환" 상태에서 beginTransition() 원칙 위반이 발생하기 쉬우므로, 예외 정책을 사전에 정의할 것.
- **로그라이크/덱빌딩 [Cycle 10]**: 턴제 덱빌딩의 핵심 재미는 "매 턴 손패 5장 중 최적 조합을 고르는 의사결정"과 "덱 성장의 빌드업 쾌감"이다. 카드 18종(공격/방어/스킬)의 조합 공간이 리플레이 가치를 만들고, 3층 맵(전투/상점/이벤트/휴식/엘리트/보스)의 경로 선택이 전략적 깊이를 더한다. 턴제라는 특성이 기술적으로 큰 이점을 준다 — setTimeout 유혹이 구조적으로 낮고, 프레임 단위 충돌 판정이 불필요하며, 모바일 터치가 카드/버튼 탭만으로 자연스럽게 작동한다. 12개 상태 머신(TITLE→MAP→PRE_BATTLE→PLAYER_TURN→ENEMY_TURN→REWARD→SHOP→REST→EVENT→VICTORY→DEFEAT)의 복잡도가 실시간 게임보다 높으므로 `enterState()` 일원화가 필수. **주의점:** 덱빌딩 밸런스(카드 시너지, 난이도 곡선)는 headless 테스트로 검증 불가 — 실플레이 데이터 수집 체계가 필요하다. 함수 시그니처 변경 시 모든 호출부 + 하위 함수 전파를 전수 검증해야 수정 회귀를 방지할 수 있다.
- **플랫포머/정밀액션 [Cycle 11 mini-platformer]**: 정밀 플랫포머의 핵심 재미는 "어려운 구간을 정밀 조작으로 돌파하는 성취감"과 "즉사→즉시 리스폰의 중독 루프"이다. 코요테 타임(6프레임), 점프 버퍼링(6프레임), 코너 보정(4px), 가변 점프 높이 등 관대한 입력 보정이 "조작이 내 의도대로 된다"는 체감을 만드는 핵심이다. AABB 충돌의 X→Y 분리 판정으로 정밀 타일맵 충돌을 달성하고, 월드별 무브셋 순차 해금(W2: 벽점프, W3: 이중점프, W4: 대시)이 점진적 학습 곡선을 만든다. lerp 카메라 + look-ahead + 흔들림 + 경계 clamping 조합의 카메라 시스템이 빠른 이동에서도 시야 안정성을 보장한다. 25개 수작업 레벨(문자열 기반)은 양적 충분성과 다양한 장애물(가시 4방향, 용암, 화염기둥, 레이저, 바람, 무너지는/이동 발판, 중력 반전 존)로 질적 다양성을 모두 달성. **주의점:** 레벨 데이터 하드코딩은 25개까지는 관리 가능하나, 그 이상이면 JSON 분리 또는 프로시저럴 생성 검토. 일일 챌린지 같은 부가 모드는 코어 메커닉 구현 후 별도 이터레이션으로 분리하는 것이 현실적이다.
- **아이들/방치형 [Cycle 11]**: 아이들 게임의 핵심 재미는 "자동화 비율을 높여가는 최적화 쾌감"과 "프레스티지로 리셋 후 더 빠른 성장을 체감하는 루프"이다. 채굴→제련→판매의 3단계 파이프라인이 자원 흐름의 시각적 이해를 돕고, 특수 업그레이드 8종 + 프레스티지 업그레이드 6종이 장기 목표를 제공한다. 아이들 장르만의 기술적 특성: (1) UI 상태(탭 전환)와 무관하게 생산 시스템이 항상 동작해야 하므로 상태×시스템 매트릭스에 "백그라운드 시스템" 개념 필요, (2) 오프라인 진행(경과 시간 × 생산율) 계산이 정확해야 함, (3) 자동 저장 주기(30초)와 프레스티지 시점 저장이 데이터 유실 방지에 핵심. 4탭 UI(생산/업그레이드/프레스티지/통계) + 스와이프 전환 + 롱프레스 연속 구매가 모바일 아이들 UX의 정석 패턴. **주의점:** 업그레이드 비용-효과 곡선의 밸런스는 headless 테스트로 검증 불가 — N회 자동 시뮬레이션 없이는 "함정 업그레이드"나 "최적 경로 고정" 문제를 잡기 어렵다. 변수 초기화 순서(let/const TDZ)에 특히 주의 — 아이들 게임은 초기화 시점에 캐시·상태·UI를 한꺼번에 세팅하므로 의존성 순서가 복잡해지기 쉽다.

- **레이싱/아케이드 [Cycle 12]**: 탑다운 드리프트 레이싱의 핵심 재미는 "핸드브레이크 → 카운터 스티어 → 코너 탈출"의 물리적 쾌감과 "드리프트 포인트 → 부스트 → 더 빠른 진입"의 리스크-리워드 순환이다. 웨이포인트 배열 기반 트랙 정의 + closestTrackPoint 도로 이탈 판정은 베지어 곡선 수준의 매끄러운 도로를 단순한 데이터로 구현하는 효율적 접근. 5개 테마 트랙(초원/사막/설산/도심/화산)의 고유 노면 효과(모래감속/빙판마찰/물웅덩이 등)가 트랙별 차별화를 만든다. 고스트 시스템(6프레임 간격 기록 → localStorage → 반투명 렌더링)은 멀티플레이어 없이 경쟁감을 추가하는 가성비 높은 패턴. 미니맵(80×80px, 트랙 윤곽 + 플레이어 위치)은 작은 구현 비용으로 공간 인지에 큰 기여. lerp 카메라 + 전방 룩어헤드 + 셰이크 감쇠가 고속 주행의 역동성을 효과적으로 연출한다. ObjectPool(연기 파티클 80개) + 타이어 자국 200개 제한으로 시각 효과와 성능의 균형 달성. **주의점:** 노면 효과를 기획할 때 "A + B" 복합 효과(속도 감소 + 조향 반전)는 절반만 구현되기 쉽다 — 세부 체크리스트로 개별 검증 필요. 트랙 데이터가 JS 배열 하드코딩이므로 10개 이상이면 JSON 분리 검토. 물리 기반 레이싱에서 가속(배속) 기능 시 서브스텝 처리 필요.
