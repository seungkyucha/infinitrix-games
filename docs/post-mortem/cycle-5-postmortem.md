# 비트 러시 (Beat Rush) — 포스트모템

## 한 줄 요약
Web Audio API로 매번 다른 음악을 절차적으로 생성하고, 4레인 노트를 타이밍에 맞춰 입력하는 리듬 아케이드 게임을 만들었다.

## 무엇을 만들었나
비트 러시는 InfiniTriX 플랫폼 최초의 리듬/음악 장르 게임이다. 외부 음악 파일 없이 Web Audio API만으로 드럼(킥/스네어/하이햇), 베이스(Am 진행), 멜로디(Am 스케일) 3트랙을 실시간 합성한다. 플레이어는 D/F/J/K 4키(또는 터치)로 판정 라인에 도달하는 노트를 입력하고, Perfect/Great/Good/Miss 4단계 판정을 받는다.

핵심 재미는 "비트와의 일체감"이다. 자신의 입력이 음악의 일부가 되는 느낌, 콤보가 쌓이면서 배경이 화려해지는 FEVER 연출, BPM이 100에서 160까지 단계적으로 올라가면서 손가락이 점점 바빠지는 긴장감이 "한 판만 더"를 만든다. 탭 노트, 롱 노트, 동시 노트 3종과 동적 밸런스 보정(HP 위기 시 밀도 감소, Miss 연속 시 격려)으로 초보자도 접근 가능하면서 고수에게는 Full Combo 도전을 제공한다.

절차적 비트맵 생성 알고리즘이 매 게임마다 다른 노트 패턴을 만들어주므로, 패턴 암기가 아닌 순수 반응 테스트로 리플레이 가치가 높다. 퍼즐(C1)→슈팅(C2)→전략(C3)→러너(C4) 이후 완전히 새로운 경험축을 플랫폼에 추가했다.

## 잘 된 점 ✅
- **4사이클 연속 재발했던 SVG 에셋 문제를 완전히 해결했다.** 기획서 §13.5의 자동 grep 검증 스크립트를 실제 도입하여 금지 패턴 7개 항목 모두 0건 PASS. "기획서 명시만으로는 불가"라는 교훈이 드디어 실행으로 이어졌다.
- **Cycle 4 B1 CRITICAL(TweenManager 경쟁 조건)을 `clearImmediate()` API 분리로 깔끔하게 해결.** resetGame()과 goToTitle()에서 즉시 정리를 사용하여 cancelAll+add 경쟁 조건을 원천 차단했다.
- **enterState() 패턴 도입으로 상태 진입 시 초기화 누락 문제를 구조적으로 해결.** Cycle 4의 타이틀 글로우 미복구 문제가 재발 불가능한 구조가 되었다.
- **기획서 §13.1 전 항목 100% 구현, 변수 사용처 전수 검증 통과.** combo 5곳, hp 4곳, inputMode 4곳 등 모든 변수가 기획 의도대로 사용되어 유령 코드 0건.
- **Web Audio API를 효과음 수준(C3~C4)에서 핵심 게임플레이 메커니즘으로 격상.** AudioContext.currentTime 기반 고정밀 비트 스케줄링과 3트랙 절차적 음악 생성이 안정적으로 동작한다.
- **코드 리뷰 2회차에서 APPROVED, 브라우저 테스트 PASS.** 1회차 지적(SVG 잔존 코드, BPM tween 이중 등록) 수정 후 신규 문제 0건으로 통과.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **1회차 리뷰에서 SVG 로딩 코드(ASSET_MAP, SPRITES, preloadAssets)가 발견되었다.** 자동 검증 스크립트를 도입했음에도 초기 코드에 잔존 패턴이 남아 있었다는 것은, 검증 스크립트를 코딩 중이 아니라 코드 완성 후에만 돌렸다는 의미. 앞으로는 코딩 단계에서도 점검이 필요하다.
- **assets/ 디렉토리에 SVG 파일 8개 + manifest.json이 여전히 잔존한다.** 코드에서 미참조이므로 기능 무영향이지만, 디스크 위생 차원에서 정리가 필요하다. 에셋 디렉토리 자동 정리 스크립트도 고려할 만하다.
- **BPM tween 이중 등록(1회차 B2)** — bpmObj에 별도 tween을 걸면서 bpm 변수 직접 반영과 중복되는 패턴이 있었다. 단일 변수에 대한 갱신 경로를 하나로 통일하는 원칙이 아직 체화되지 않았다.
- **절차적 음악의 청감 품질은 "듣기에 나쁘지 않은" 수준.** Am 마이너 키 고정과 낮은 볼륨(0.3)으로 불쾌감은 피했지만, 곡의 다양성이나 감성적 몰입은 제한적이다. 코드 진행 변화나 리버브 효과를 추가하면 한 단계 올릴 수 있다.
- **모바일 터치 레이턴시 보정(+15ms 오프셋)의 실제 효과를 측정하지 못했다.** 실기기 테스트 없이 Puppeteer headless만으로는 터치 판정 체감을 검증할 수 없다.

## 기술 하이라이트 🛠️
이번 게임의 기술적 핵심은 **AudioContext.currentTime 기반 비트 스케줄링**이다. setTimeout/setInterval을 완전히 배제하고, Web Audio API의 고정밀 오디오 클럭으로 비트 타이밍을 계산한다. `scheduleAhead = 0.1초` 미리 스케줄링 + `lookAhead = 0.05초` 주기 체크로 오디오와 게임 로직의 정확한 동기화를 달성했다.

**절차적 비트맵 생성 알고리즘**도 주목할 만하다. 난이도 레벨(0~10)에 따라 사용 레인 수, 노트 유형(탭/롱/동시), 밀도를 점진적으로 조절하면서, "4레인 동시 금지", "롱 노트 중 같은 레인 탭 금지", "연속 3노트 후 1비트 휴식" 같은 안전 규칙으로 물리적으로 불가능한 패턴을 방지한다.

**offscreen canvas 격자 그리드 캐시**로 매 프레임 재드로잉을 방지하고, globalAlpha 조절만으로 비트 반응형 배경을 구현한 것도 성능 최적화의 좋은 사례다. ObjectPool 4종(노트 40, 파티클 60, 파동 10, 롱 노트 10)으로 GC 스파이크 없는 부드러운 60fps를 유지한다.

## 다음 사이클 제안 🚀
1. **물리 기반 장르 도전 (예: 미니 골프, 앵그리버드 스타일 투사체)** — 5사이클간 쌓은 Canvas + TweenManager + ObjectPool 인프라 위에 간단한 2D 물리(중력, 반사, 마찰)를 얹으면 새로운 경험축이 열린다. 물리 엔진 없이 기본 벡터 연산만으로 충분히 구현 가능한 규모로.
2. **멀티플레이어 또는 비동기 경쟁 요소** — 현재 모든 게임이 솔로 플레이. localStorage에 "오늘의 시드"를 저장해서 같은 시드로 플레이한 점수를 비교하는 일일 챌린지 모드를 추가하면, 외부 서버 없이도 경쟁감을 줄 수 있다.
3. **공용 인프라 모듈 분리** — TweenManager(clearImmediate 포함), ObjectPool, listen()+destroy(), Web Audio SFX, TransitionGuard가 5사이클간 안정화되었다. 이를 `shared/engine.js`로 분리하면 새 게임 시작 시 보일러플레이트를 대폭 줄이고 버그 수정이 전 게임에 반영된다.
