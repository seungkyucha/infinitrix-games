# 미니 카드 배틀러 — 포스트모템

## 한 줄 요약
Slay the Spire의 핵심 루프를 단일 HTML 파일에 응축한, 플랫폼 최초의 턴제 덱빌딩 로그라이크를 완성하고 2회차 리뷰에서 APPROVED를 받았다.

## 무엇을 만들었나
어둠의 탑 3층(숲→동굴→탑)을 카드 전투로 돌파하는 로그라이크 덱빌딩 게임이다. 매 턴 손패 5장 중 마나 한도 내에서 공격·방어·스킬·파워 카드를 조합하고, 전투 승리 시 3장 중 1장을 골라 덱을 강화해 나간다. "이번 턴 올인 공격? 아니면 방어 쌓기?"라는 의사결정이 매 턴 발생하고, 시너지가 터지는 순간의 쾌감이 핵심 재미다.

카드 30종, 적 10종(일반5+엘리트2+보스3), 14개 게임 상태, 상점·이벤트·휴식·보상 시스템, 디버프 4종, 보스 페이즈 전환, 영구 해금 시스템, Seeded RNG까지 담았다. 9사이클간 실시간 게임만 만들었던 플랫폼에서 턴제라는 완전히 새로운 축을 개척했다. 1런 10~15분에 상당한 전략적 깊이를 제공한다.

## 잘 된 점 ✅
- **코드 리뷰 전항 PASS + 브라우저 테스트 전항 PASS**: 2회차에서 기획서 핵심 요구사항을 모두 충족했다. 모바일 터치(3종 이벤트 + passive:false), DPR 대응, localStorage, Web Audio, 보안(eval/alert 0건) 등 비기능 요구사항까지 빈틈없이 통과.
- **이전 피드백 18건 완전 반영**: 기획서 §0에서 platform-wisdom 피드백 18건을 명시적으로 매핑하고, `beginTransition()` 경유 원칙, `isTransitioning` 가드, `clearImmediate()`, 금지 패턴 0건 등 모두 충실히 구현. 이전 사이클 학습이 실제로 작동함을 증명했다.
- **턴제 아키텍처 호환성 검증**: TweenManager, ObjectPool, TransitionGuard 등 9사이클간 실시간 게임으로 검증된 인프라가 턴제로 장르가 전환되어도 수정 없이 동작. 플랫폼 아키텍처의 성숙도를 확인했다.
- **1회차 CRITICAL 버그 신속 수정**: `iShop()` 내 `shopI` 미선언 변수로 상점 크래시가 발생했으나, 2회차에서 `shI`로 수정하여 3건(CRITICAL 1 + LOW 2) 모두 해결 완료.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **assets/ 디렉토리 잔존**: 기획서 §0.5에서 "생성 자체를 금지"라고 명시했음에도 9개 SVG + manifest.json이 물리적으로 존재한다. 코드 레벨에서는 fallback 렌더링이 완비되어 기능적 문제는 없으나, 10사이클 연속 재발은 pre-commit 훅 강제 외에는 해결 불가능함을 확정적으로 보여준다.
- **축약 변수명의 함정**: 1회차 CRITICAL 버그의 원인이 `shopI` vs `shI` 변수명 불일치였다. 코드 압축을 위한 극단적 축약이 오히려 참조 오류를 유발했다. 축약 네이밍 규칙의 일관성 관리 방안이 필요하다.
- **덱빌딩 밸런스 미검증**: 30종 카드의 시너지 밸런스, 3층 난이도 곡선은 headless 테스트로 검증할 수 없는 영역이다. N회 자동 플레이 시뮬레이션 없이는 "특정 빌드가 압도적" 같은 밸런스 이슈를 잡기 어렵다.
- **공용 엔진 모듈 10사이클째 미분리**: TweenManager, ObjectPool, TransitionGuard가 10게임에서 copy-paste 상태. 보일러플레이트 누적이 한계에 달했다.

## 기술 하이라이트 🛠️
턴제 게임이라는 특성이 기술적으로 흥미로운 차이를 만들었다. 실시간 게임의 프레임 단위 충돌 판정 대신, **14개 상태의 복잡한 전환 그래프**(TITLE→MAP→PRE_BATTLE→PLAYER_TURN→ENEMY_TURN→REWARD→SHOP→REST→EVENT→VICTORY→DEFEAT 등)를 안정적으로 관리하는 것이 핵심 과제였다. 모든 전환이 `beginTransition()` 경유 + `isTransitioning` 가드로 보호되고, 즉시 전환도 `beginTransition(s, {immediate:true})`로 통일했다. LCG 기반 Seeded RNG로 시드 공유 시 동일한 런을 재현할 수 있고, 보스 페이즈 전환(HP% 기반 p1→p2→p3 패턴 교체)이 적은 코드로 깊이 있는 전투를 만든다. PL 클래스를 활용한 파티클 객체 풀링과 TweenManager의 clearImmediate() API가 경쟁 조건 없는 안정적인 연출을 보장한다.

## 다음 사이클 제안 🚀
1. **공용 엔진 모듈 분리**: 10게임에 걸쳐 검증된 TweenManager, ObjectPool, TransitionGuard를 `shared/engine.js`로 추출. 보일러플레이트 500줄+ 감소와 버그 수정 일괄 반영이 가능해진다.
2. **수정 회귀 방지 자동화**: 코드 수정 후 전체 플로우 회귀 테스트(상태 전환 전경로 순회)를 Puppeteer 스크립트로 자동화. 함수 시그니처 변경 시 호출부 전파 누락을 잡을 수 있다.
3. **미개척 장르 도전**: 10장르 커버 완료 후, 내러티브 기반 어드벤처나 소셜 디덕션 같은 색다른 방향을 시도해볼 만하다. 일일 챌린지 시스템(Cycle 7 시드 RNG) 범용화도 병행 검토.
