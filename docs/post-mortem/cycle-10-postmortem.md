# 미니 카드 크롤러 — 포스트모템

## 한 줄 요약
Slay the Spire의 핵심 재미를 단일 HTML 파일에 응축한 턴제 덱빌딩 로그라이크를 만들었다.

## 무엇을 만들었나
어둠의 탑 3층을 카드 전투로 돌파하는 로그라이크 덱빌딩 게임이다. 매 턴 손패 5장 중 마나 한도 내에서 공격·방어·스킬 카드를 조합하고, 전투 승리 시 3장 중 1장을 골라 덱을 강화해 나간다. "이번 턴 올인 공격? 아니면 방어 쌓기?"라는 의사결정이 매 턴 발생하고, 덱이 점점 강해지면서 시너지가 터지는 순간의 유레카 경험이 핵심 재미다.

맵은 3층 구조(숲→동굴→탑 꼭대기)로 전투·상점·이벤트·휴식·엘리트·보스 노드가 분기 경로로 연결된다. 카드 18종, 적 7종(보스 포함), 12개 게임 상태, 상점 시스템, 카드 보상, 이벤트 2지선다까지 — 1런 10~15분에 상당한 깊이를 담아냈다. 플랫폼 최초의 턴제 게임이자 카드/덱빌딩 장르 첫 도전이라는 점에서 의미가 크다.

## 잘 된 점 ✅
- **플랫폼 장르 공백 해소**: 9사이클간 실시간 게임만 만들었는데, 턴제 + 카드/덱빌딩이라는 완전히 새로운 축을 개척했다. strategy 장르 보강과 easy 난이도 확대도 동시에 달성.
- **기능 완성도 10/10 PASS**: 리뷰 체크리스트 전항 통과. 12개 상태 전환, 모바일 터치 3종 + 44px+ 터치 영역, DPR 대응, localStorage 저장/로드, 파티클 풀링까지 빈틈없이 구현됐다.
- **100% Canvas 코드 드로잉 + 에셋 위반 0건**: 8사이클 연속 재발했던 assets/ 문제가 드디어 깔끔하게 해결됐다. thumbnail.svg만 남긴 채 모든 비주얼을 코드로 그렸다.
- **금지 패턴 완전 클린**: eval, alert, setTimeout 상태 전환, innerHTML 등 6개 금지 패턴 검출 0건. 보안·안정성 모두 만점.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **3회차 리뷰까지 소요**: 1회차에서 MINOR 4건(에너지 체크, dead code, assets 위반, 버튼 크기), 2회차에서 수정 과정 중 `timestamp` ReferenceError CRITICAL이 신규 도입되어 3회차까지 갔다. 수정 시 새 버그를 도입하지 않는 회귀 검증 프로세스가 아직 부족하다.
- **기획서 game-id 불일치**: 기획서에는 `mini-card-battler`로, 실제 구현은 `mini-card-crawler`로 배포됐다. 기획→구현 과정에서 네이밍이 변경되었으나 기획서 갱신이 누락된 점은 관리 개선이 필요하다.
- **덱빌딩 밸런스 미검증**: 18종 카드의 시너지 밸런스, 3층 난이도 곡선 등은 headless 테스트로 검증할 수 없는 영역이다. 실플레이 데이터 없이는 "특정 카드 조합이 압도적" 같은 밸런스 이슈를 잡기 어렵다.
- **공용 엔진 모듈 미분리**: TweenManager, ObjectPool, TransitionGuard 등이 10게임에서 copy-paste 상태. 보일러플레이트가 계속 누적되고 있다.

## 기술 하이라이트 🛠️
턴제 게임이라는 특성이 기술적으로 흥미로운 차이를 만들었다. 실시간 게임에서 필수적인 프레임 단위 충돌 판정·물리 시뮬레이션이 불필요하고, 대신 **상태 머신 12개의 복잡한 전환 그래프**(TITLE→MAP→PRE_BATTLE→PLAYER_TURN→ENEMY_TURN→REWARD→SHOP→REST→EVENT→VICTORY→DEFEAT)를 안정적으로 관리하는 것이 핵심 과제였다. `enterState()` 일원화 패턴과 `render(ctx, dt, timestamp)` 시그니처 통일로 모든 상태에서 일관된 렌더링을 보장했다. 파티클 풀 재사용, `dt = Math.min(delta, 0.05)` 프레임 캡, `destroyListeners()` 패턴 등 이전 사이클에서 검증된 인프라가 장르가 완전히 달라져도 그대로 작동했다는 점이 플랫폼 아키텍처의 성숙도를 보여준다.
