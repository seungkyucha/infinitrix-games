# 프리즘 브레이크 (Prism Break) — 포스트모템

## 한 줄 요약
빛 굴절 퍼즐 + 실시간 그림자 적 전투 하이브리드를 시도했으나, TDZ 크래시로 게임이 한 번도 실행되지 못한 채 NEEDS_MAJOR_FIX로 마무리된 사이클.

## 무엇을 만들었나
프리즘 브레이크는 크리스탈을 12×8 그리드에 배치·회전하여 빛 경로를 설계하고, 실시간으로 접근하는 그림자 적을 소멸시키는 액션 퍼즐이다. 반사·분산·집속·색상 필터·시간·폭발 등 7종 크리스탈과 5종 그림자 적의 조합이 핵심 재미이며, BFS 기반 빛 경로 추적과 스테이지 해법 검증이 퍼즐의 공정성을 보장하는 구조다.

기획서는 역대 최고 수준의 완성도를 보였다. F1~F30까지 30건의 피드백 매핑, 5구역×3스테이지+보스 3종의 Phase 1 MVP, DDA 4단계 난이도 곡선, 업그레이드 트리 3갈래×5단계, ko/en 다국어 지원까지 세밀하게 설계되었다. 코드 정적 분석에서도 INIT_EMPTY 패턴, SeededRNG 완전 사용(Math.random 0건), 프로시저럴 SFX+BGM 12종, BFS 스테이지 검증 등 플랫폼의 누적 교훈이 충실히 반영되어 있었다.

그러나 이 모든 것이 단 하나의 P0 버그 — `onResize()` 콜백에서 초기화 미완료 상태의 `engine` 변수를 참조하는 TDZ(Temporal Dead Zone) 크래시 — 때문에 빛을 보지 못했다. 2차 리뷰까지 진행했으나 P0이 미수정된 채로 제출되어, 검은 화면만 표시되고 모든 브라우저 테스트가 FAIL로 기록되었다.

## 잘 된 점 ✅
- **assets/ 문제 완전 해결**: 1차 리뷰에서 지적된 42개 에셋 파일을 2차에서 전량 삭제하고, 모든 렌더링을 프로시저럴 Canvas 도형으로 전환했다. 코드 곳곳에 `// F1/F24: no assets` 주석으로 의도를 명시한 점도 좋았다.
- **코드 아키텍처 우수**: TRANSITION_TABLE 4상태 단일 정의, INIT_EMPTY 패턴, SeededRNG 완전 사용, bossRewardGiven 가드 플래그, beginTransition proxy 패턴 등 38사이클간 누적된 모범 사례가 빠짐없이 적용되었다.
- **기획서 충실도**: 크리스탈 7종, 적 5종, 보스 3종, BFS 경로 검증, DDA 4단계, 다국어 지원 등 기획 명세 대부분이 코드에 정확히 반영되었다(정적 분석 기준).
- **프로시저럴 사운드**: Web Audio API로 12종 SFX + 4종 BGM을 완전히 절차적으로 생성하여 외부 에셋 0건 원칙을 유지했다.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **P0 TDZ 미수정으로 게임 완전 불능**: 1차 리뷰에서 명확한 수정 방법(파라미터 전달 + 가드 추가)까지 제시되었으나, 2차에서도 동일 상태로 재제출되었다. `onResize(w, h)` → `initBgParticles(count, w, h)` + `calculateGridLayout(w, h)` 변경과 `if (typeof engine === 'undefined') return;` 가드 5줄이면 해결되는 문제였다.
- **TDZ 패턴 재발**: Cycle 5~38까지 F12(INIT_EMPTY 패턴)로 방어해왔고, Cycle 31에서도 동일 계열 크래시가 발생한 전력이 있다. Engine 생성자 내 콜백 호출이라는 변형 패턴을 기획서에서 커버하지 못했다.
- **모바일 터치 타겟 미달**: `Math.max(30, G.cellSize)`로 최소 30px만 보장하여 48px 기준에 미달. F11(터치 타겟 48px) 위반이 12사이클째 반복 중이다.
- **fadeAlpha 동기화 불완전**: syncFade/syncOut 트윈이 `_t` 값만 변경하고 `G.fadeAlpha`에 실제 반영하지 않는 구조적 결함이 미수정.
- **런타임 검증 완전 불가**: P0 차단으로 게임플레이, 비주얼, 밸런스 등 모든 실질 검증이 이루어지지 못했다. 코드가 "잘 작성되었을 가능성이 높다"는 정적 분석 결과만 남았다.

## 기술 하이라이트 🛠️
- **BFS 빛 경로 추적**: `traceLight()` 함수가 BFS 기반으로 광원→크리스탈→크리스탈→목표 경로를 탐색하며, `validateStageReachability()`가 스테이지 생성 시점에 해법 존재를 보장한다. 빛 굴절 퍼즐의 공정성을 알고리즘적으로 담보하는 설계.
- **프로시저럴 크리스탈 렌더링**: 7종 크리스탈, 5종 적, 3종 보스를 모두 Canvas 2D API 도형으로 구현. 외부 이미지 없이도 빛 굴절과 프리즘 테마를 시각적으로 표현한 접근.
- **DDA 4단계 동적 난이도**: 플레이어 성과에 따라 적 스폰 밀도, 이동 속도, 안전 구간을 4단계로 자동 조절하는 시스템. 수학 공식 기반으로 밸런스 곡선이 설계되었다.
- **TDZ의 변형 패턴**: `const engine = new Engine(...)` → Engine 생성자 → `resize()` → `onResize()` → `engine.W` 참조라는 "생성자 내 콜백이 미완료 변수를 참조"하는 TDZ 패턴은 기존 F12(INIT_EMPTY)로는 방지되지 않는 새로운 변형이다. 이를 platform-wisdom에 추가하여 향후 재발을 방지해야 한다.

## 다음 사이클 제안 🚀
1. **Engine 생성자 콜백 TDZ 방어 패턴 표준화**: `onResize(w, h)` 콜백에서 `engine` 직접 참조 금지를 F12 확장 규칙으로 추가. Engine 초기화 완료 전 호출되는 모든 콜백에서 파라미터만 사용하도록 강제하는 코딩 가이드라인 수립.
2. **Puppeteer 스모크 테스트에 "엔진 초기화 성공" 검증 추가**: 기존 "index.html 존재 + 페이지 로드 + 콘솔 에러 0건" 3단계에 "engine 객체 접근 가능 + G.state 전환 발생" 검증을 추가하여 TDZ 크래시를 자동으로 탐지.
3. **빛 굴절 퍼즐 재도전**: 프리즘 브레이크의 기획과 코드 품질은 P0만 수정하면 즉시 플레이 가능한 수준이었다. P0 5줄 수정 후 3차 리뷰를 진행하거나, 동일 기획을 다음 사이클에서 빠르게 재구현하는 것을 추천한다.
