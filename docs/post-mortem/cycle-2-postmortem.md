# 스타 가디언 — 포스트모템

## 한 줄 요약
네온 우주 종스크롤 슈팅을 만들며 TweenManager·ObjectPool 인프라를 검증했지만, 정작 그 Tween 시스템의 상태별 업데이트 누락으로 핵심 UI가 보이지 않는 아이러니를 경험했다.

## 무엇을 만들었나
**스타 가디언**은 우주 배경에서 플레이어 전투기를 조종해 웨이브별로 밀려오는 적 함대를 격파하는 종스크롤 슈팅 게임이다. 스카우트·파이터·탱크·다트 4종의 적과 3페이즈 순환 보스전, 무기 강화(4단계)·HP 회복·쉴드 3종 파워업, 1초 이내 연속 처치 콤보 시스템, 5단계 점수 이정표까지 — 아케이드 슈팅의 핵심 재미 요소를 단일 HTML 파일 1,185줄에 담았다.

3레이어 패럴랙스 별 배경 위에 네온 글로우 이펙트, 스크린 셰이크, 폭발 파티클이 어우러져 시각적 몰입감이 높다. 키보드(WASD+방향키)·마우스(lerp 추적)·터치(가상 조이스틱+오토파이어) 3종 입력을 모두 지원하며, 한 판 3~7분의 짧고 강렬한 세션으로 "한 판 더"의 중독성을 노렸다. Cycle 1의 퍼즐에서 완전히 다른 기술 영역(실시간 입력, 충돌 감지, 프레임 기반 루프)에 도전한 의미 있는 사이클이었다.

## 잘 된 점 ✅
- **Cycle 1의 4대 문제 중 3개를 완벽히 해결했다.** Google Fonts 제거(M1), 미사용 에셋 정리(M2), TweenManager로 애니메이션 시스템 구축(M4). 기획서 §0에 이전 사이클 피드백을 명시적 테이블로 정리한 전략이 효과적이었다.
- **TweenManager + ObjectPool 아키텍처가 실전에서 검증되었다.** Tween은 적 등장·폭발·파워업·웨이브 텍스트·HP바·피격 플래시 등 16곳 이상에서 활용되었고, ObjectPool은 총알·적·파티클 등 6종 엔티티의 GC 압박을 제거했다. 이 두 시스템은 플랫폼의 표준 인프라로 자리잡을 수 있다.
- **기획서 충실도가 매우 높다.** 적 4종+보스의 이동/공격 패턴, 파워업 드롭 확률, 웨이브 스케일링 공식까지 기획서에 HEX·수식을 명시한 전략이 다시 한번 효과를 증명했다. 20개 기능 체크리스트 중 18개 완전 구현.
- **코드 규모 2배 성장을 소화했다.** Cycle 1(~600줄)에서 Cycle 2(~1,185줄)로 복잡도가 크게 올랐지만, 상태 머신 6상태·객체 풀링·tween 시스템으로 구조적으로 관리되었다.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **[B1/B2] CONFIRM_MODAL·PAUSE 상태에서 Tween이 업데이트되지 않아 Canvas 모달이 투명(alpha=0) 상태로 고정된다.** Cycle 1 M3(confirm() 대체)를 해결하려고 Canvas 모달을 구현했지만, `loop()` 함수의 해당 상태 분기에서 `tw.update(ms)`를 호출하지 않아 tween 애니메이션이 진행되지 않았다. R키를 눌러도 모달이 보이지 않는 치명적 UX 결함. 수정 자체는 각 1줄 추가로 해결 가능하지만, "상태별로 어떤 시스템이 돌아야 하는지"를 설계 시점에 매트릭스로 정리했어야 했다.
- **[B4] NEW BEST 판정이 항상 false.** `saveBest()` 호출 후에 `score > getBest()`를 비교하므로 이미 갱신된 값과 비교되어 신기록 연출이 영원히 표시되지 않는다. 2줄 순서 변경으로 해결 가능한 논리 오류.
- **[B3] 게임오버 전환에 setTimeout(600) 잔존.** Cycle 1 M4에서 지적된 패턴의 재사용. tween의 onComplete 콜백으로 대체해야 한다.
- **[B5] 적 SVG에 source-atop 합성 시 배경 사각형 노출.** feGaussianBlur 필터가 viewBox 전체에 반투명 픽셀을 생성하는 문제. 시각적으로 거슬리지만 게임플레이에는 영향 없다.

## 기술 하이라이트 🛠️
이번 사이클의 가장 큰 기술적 성과는 **재사용 가능한 게임 인프라 2종의 실전 검증**이다.

**TweenManager**는 `add(target, props, duration, easing, onComplete)` API 하나로 어떤 객체든 속성 보간 애니메이션을 걸 수 있다. 이징 함수 5종(linear, easeOutQuad, easeInQuad, easeOutBack, easeOutElastic)을 내장하며, Cycle 1의 최대 아쉬움이었던 "움직임 없는 게임"을 근본적으로 해결했다. 다만 이번에 배운 교훈은 — 시스템을 만들었어도 **모든 상태에서 업데이트를 호출하지 않으면 무용지물**이라는 것이다.

**ObjectPool**은 acquire/release 패턴으로 총알·적·파티클 등 6종 엔티티를 풀링한다. 매 프레임 수십 개의 객체가 생성/소멸되는 슈팅 게임에서 GC 스파이크 없이 60fps를 유지할 수 있었다. **보스 3페이즈 AI**(직선 연사→원형 확산→추적탄, HP 50% 이하 가속)도 상태 머신 위에 상태 머신을 얹는 패턴으로 깔끔하게 구현되었다.

AABB 충돌 감지에서 플레이어 히트박스를 시각 크기(40×48)보다 작게(24×24) 설정한 "관대한 판정"도 좋은 선택이었다. 체감 난이도가 적절해져 "운 나쁜 사망"이 줄었다.
