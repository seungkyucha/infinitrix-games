# 아케인 바스티온 — 포스트모템

## 한 줄 요약
로그라이크 포지 디펜스를 기획했으나, 2회차 리뷰까지 index.html이 생성되지 않아 **구현 0%, 게임 실행 불가** 상태로 NEEDS_MAJOR_FIX 판정을 받았다.

## 무엇을 만들었나
마법사가 바스티온(요새)을 중앙에 두고, 사방에서 몰려오는 적을 직접 주문으로 처치하면서 마법 타워를 배치하고, 웨이브 사이마다 로그라이크 3택 1 업그레이드로 성장하는 **이중 전투 + 타워 디펜스 + 로그라이크** 하이브리드 게임을 목표로 했다. 15웨이브(3바이옴: 숲→동굴→화산) + 보스 3체 + 적 5종 + 타워 3종이라는 야심 찬 기획이었다.

그러나 **실제 구현은 전무하다.** 2회차 리뷰 시점에서도 `index.html`이 존재하지 않고, 오히려 기획서에서 명시적으로 금지한 `assets/` 디렉토리가 SVG 파일 8개와 함께 생성되어 F1 규칙을 정면 위반하는 **퇴보**가 발생했다. 기획서 §1~§13의 어떤 기능도 코드로 작성되지 않은 상태다.

## 잘 된 점 ✅
- **기획서 품질은 역대 최고 수준.** F1~F24까지 16사이클 누적 교훈을 전수 반영한 매핑 테이블, 5상태×6시스템 매트릭스, 웨이브별 밸런스 테이블, 입력 3종 매핑, 수치 정합성 검증 체크리스트 등 기획 문서 자체의 완성도는 뛰어나다.
- **장르 균형 분석이 전략적.** 플랫폼에서 가장 부족한 action(21.1%)과 strategy(26.3%)를 동시에 보강하겠다는 명확한 목적 의식이 있었다.
- **이전 사이클(네온 브릭 브레이커)에서 APPROVED를 달성한 직후의 시도.** 검증된 워크플로우와 기술 인프라가 축적된 상태에서의 도전이었다.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **가장 기본적인 산출물(index.html)이 2회 연속 미생성.** 기획서가 아무리 정교해도 실제 코드가 없으면 의미가 없다. 리뷰 전 "파일 존재 여부" 스모크 테스트가 여전히 자동화되지 않은 것이 근본 원인이다.
- **F1 규칙(assets/ 금지) 17사이클째 재발, 이번에는 역방향 퇴보.** 1회차에서 없던 assets/ 디렉토리가 2회차에 새로 생성됨. SVG 에셋 8개(bg-layer, enemy, player, powerup, ui-heart 등)가 포함되어 "100% Canvas 코드 드로잉" 원칙을 정면 위반했다.
- **기획 범위가 여전히 과대했을 가능성.** 15웨이브×5적종×3타워×3보스×3바이옴×로그라이크 업그레이드 시스템은 단일 사이클 구현에 비현실적일 수 있다. Cycle 16에서 "범위 축소 전략"을 강조했음에도 불구하고, 이번 기획은 action+strategy 하이브리드라는 높은 복잡도를 택했다.
- **시간이 있었다면:** 최소한 TITLE→PLAYING→GAMEOVER 3상태 + 적 1종 + 타워 0개의 MVP를 먼저 구현하고, 점진적으로 확장하는 전략이 필요했다.

## 기술 하이라이트 🛠️
실제 구현 코드가 없어 기술 하이라이트를 기록할 수 없다. 다만 기획서에서 설계한 기술적 접근은 주목할 만하다:
- **offscreen canvas 바이옴 배경 3종 캐싱(F10):** 숲/동굴/화산 배경을 각각 offscreen canvas에 사전 렌더링하고, 바이옴 전환 시 교체하는 설계
- **3중 가드 체계(F5):** `waveClearing`/`isTransitioning`/`isBossActive` 플래그로 웨이브→보스→업그레이드 전환의 경쟁 조건 방지
- **순수 함수 기반 전체 게임 로직(F3):** §10에 정의된 모든 함수 시그니처가 파라미터 기반, 전역 직접 참조 0건 목표
- 이 설계들은 다음 시도에서 그대로 재활용할 가치가 있다.

## 다음 사이클 제안 🚀
1. **아케인 바스티온 재도전 시 MVP 우선 전략:** 상태 3개(TITLE/PLAYING/GAMEOVER) + 적 1종(슬라임) + 타워 0개 + 웨이브 3개로 시작하여 "실행 가능한 게임"을 먼저 확보한 후, 기능을 점진적으로 추가. 기획서는 이미 완성되어 있으므로 구현에만 집중.
2. **CI/pre-commit 훅 즉시 등록:** `index.html 존재` + `assets/ 미존재` 검증을 리뷰 제출 전 자동화. 이번 사이클에서 같은 문제가 2회 연속 발생한 것은 수동 검증의 한계를 명확히 보여준다.
3. **범위를 더 줄인 대안 장르 시도:** action+strategy 하이브리드 대신, 상태 4개 이내의 단일 장르 게임(솔리테어, 타이핑 등)으로 "초회 APPROVED" 연속 달성에 집중하는 것도 전략적 선택이다.
