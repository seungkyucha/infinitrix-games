# 폴리 스파이어 — 포스트모템

## 한 줄 요약
Slay the Spire에서 영감받은 로우폴리 3D 턴제 덱빌더를 단일 HTML 파일로 구현하여, 30장 카드 풀·5유물·3층 보스 런 구조를 완성했다.

## 무엇을 만들었나
**폴리 스파이어**는 로우폴리 3D 아트 스타일의 로그라이크 덱빌더다. 플레이어는 기본 덱 10장으로 시작해 3개 층의 노드 맵을 탐험하며 카드를 모으고, 턴제 전투로 보스를 쓰러뜨린다. 전투·상점·휴식·이벤트 5종 노드가 매 런마다 다른 경로를 제공하고, 카드 보상 "3장 중 1장 선택"이 모든 전투를 의미 있는 전략적 결정으로 만든다.

카드 풀 30장(공격 12·방어 9·스킬 9), 유물 5종, 포션 4종, 적 6종+보스 3종으로 단일 HTML 파일 안에서 놀라운 전략 깊이를 달성했다. 각 보스는 HP 50% 기준 2페이즈로 전환되어, "거의 잡았다"에서 역전당하는 긴장감이 핵심 재미를 견인한다. 3단계 난이도와 등급(S/A/B/C) 시스템이 반복 플레이 동기를 부여한다.

## 잘 된 점 ✅
- **이전 사이클 학습의 체계적 반영**: 기획서에 12항목 체크리스트를 배치해 thumbnail 누락, 에셋 스타일 불일치, 엔진 API 오용 등 이전 이슈를 사전 차단했다. 리뷰 A~H 전 영역 PASS라는 결과가 이를 증명한다.
- **카드 데이터 선언적 설계**: 30장 카드를 JSON 데이터로 정의하고 효과를 선언적으로 기술한 덕분에, 밸런스 조정과 카드 추가가 데이터 수정만으로 가능한 확장성 높은 구조를 만들었다.
- **IX 엔진 100% 준수**: 자체 addEventListener/setTimeout 사용 0건, 3회 연속 재시작 JS 에러 0건. 3사이클 연속으로 엔진 호환성 PASS를 달성하며 안정적인 패턴을 확립했다.
- **모바일·데스크톱 동시 대응**: isMobileView/safeArea/isPortrait 분기를 10회 이상 사용하고, 세로·가로 레이아웃을 별도 설계해 터치 전용 플로우를 완성했다.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **3차 리뷰까지 소요 (3사이클 연속)**: 1차에서 TDZ 에러로 게임이 아예 실행되지 않았고, 2차에서 BATTLE 첫 턴 카드/적 버튼이 생성되지 않는 치명적 버그가 발견됐다. 기본적인 "로드→1회 통플레이→재시작" 셀프 QA만 했어도 1차에서 잡을 수 있었던 이슈들이다.
- **모바일 적 타겟 버튼 크기 미달**: 390px 뷰포트에서 적 타겟 버튼 너비가 41px로 IX.Button 권장 44px에 미달. 기능 동작에는 영향이 없지만, 동적 크기 버튼에 `Math.max(44, ...)` 가드를 표준 적용해야 한다.
- **카드 업그레이드 시스템 아쉬움**: 휴식 노드에서 카드 강화가 가능하나, 단순 수치 증가에 그쳤다. 업그레이드 시 부가 효과를 추가하는 방식이었다면 빌드 분기가 훨씬 풍부했을 것이다.
- **이벤트 다양성 부족**: 이벤트 노드가 존재하지만 종류가 제한적이다. 시간이 더 있었다면 리스크-리워드 선택이 극적인 이벤트를 5~6종 추가하고 싶었다.

## 기술 하이라이트 🛠️
**턴제 전투 상태 머신**: 플레이어→적→턴시작 순서를 엄격히 제어하는 상태 머신을 구축했다. 독/버프/디버프가 턴 경계에서 정확히 적용되며, 카드 사용→에너지 차감→데미지 적용→사망 체크→보상 분기의 복잡한 흐름을 안정적으로 처리한다.

**rebuildBattleButtons() 동적 UI 패턴**: 카드 사용·적 사망·턴 전환 등 전투 상태가 변할 때마다 IX.Button을 재구성하는 패턴을 확립했다. 2차 리뷰에서 enter() 시점 누락 버그로 발견되었지만, 수정 후 동적 UI의 표준 패턴이 되었다.

**노드 맵 절차적 생성**: Slay the Spire 스타일로 층당 6~8노드의 분기 경로를 절차적으로 생성한다. 전투·엘리트·상점·휴식·이벤트·보스 노드가 규칙에 따라 배치되어 매 런마다 다른 경험을 보장한다.

## 다음 사이클 제안 🚀
1. **퍼즐/매치3 장르 도전**: 턴제(Cycle 1)·실시간(Cycle 2)·카드(Cycle 3) 장르를 커버했으니, 퍼즐 계열로 범위를 확장하자. 스코프가 상대적으로 작아 1차 리뷰 통과 목표와도 부합한다.
2. **셀프 QA 프로토콜 의무화**: "로드→1회 통플레이→재시작 1회" 체크를 코드 제출 전 필수 단계로 강제하자. 3사이클 연속 3차 리뷰라는 비효율적 패턴을 끊어야 한다.
3. **동적 UI 버튼 크기 가드 표준화**: `Math.max(44, ...)` min-width 제약을 동적 버튼 생성의 기본 패턴으로 정착시켜, 모바일 사용성 이슈를 구조적으로 방지하자.
