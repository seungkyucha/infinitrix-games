# 컬러 머지 퍼즐 — 포스트모템

## 한 줄 요약
InfiniTriX 플랫폼의 첫 게임으로, 2048 스타일의 5×5 색상 머지 퍼즐을 단일 HTML 파일로 완성했다.

## 무엇을 만들었나
같은 색상의 블록을 합쳐 빨강에서 보라까지 7단계 무지개 색상을 진화시키는 퍼즐 게임이다. 2048의 슬라이드 머지 메카닉을 기반으로 하되, **무지개 색상 진화 시스템**과 **콤보 체인 배수**를 추가해 시각적 쾌감과 전략적 깊이를 동시에 노렸다. 보라 두 개를 합치면 무지개 블록이 생성되며 보드를 정리해주는 보너스 이벤트가 발생한다.

기술적으로는 외부 프레임워크 없이 **Vanilla JS + Canvas API** 단일 파일(1,111줄)로 구현했다. 키보드(방향키/WASD), 터치 스와이프, 마우스 드래그 3종 입력을 모두 지원하며, DPR 대응과 반응형 레이아웃으로 모바일부터 데스크톱까지 커버한다. 점수별로 블록 생성 분포와 추가 스폰 확률이 변하는 동적 난이도 시스템도 탑재했다.

## 잘 된 점 ✅
- **기획 충실도 92%** — 24개 기획 항목 중 22개가 완벽히 일치. 색상 HEX 코드, 머지 점수 테이블, 콤보 공식까지 기획서 그대로 구현됨
- **게임 상태 머신(LOADING→TITLE→PLAYING→GAMEOVER)** 설계가 명확해 상태 전환 버그 없이 깔끔하게 동작
- **Canvas 기반 순수 렌더링** — DOM 접근 제로, 파티클/floatingText 수명 관리까지 구현되어 60fps 안정적 유지
- **보안/안전성 만점** — eval, alert/confirm, XSS 경로 없음. localStorage도 try-catch로 감싸 iframe sandbox 환경에서도 안전
- **SVG 커스텀 에셋 + 폴백 렌더링** — 에셋 로딩 실패 시에도 코드 기반 렌더링으로 게임이 정상 동작하는 방어적 설계

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **블록 이동 슬라이드 애니메이션 미구현 [M4]** — `setTimeout(160)`으로 입력 잠금만 걸고 즉시 위치 변경. 기획서의 "150ms ease-out 슬라이드"가 빠져 시각적 피드백이 부족하다. 스폰/머지 이펙트는 잘 되어 있어 대조적으로 더 눈에 띈다.
- **불필요한 에셋 3개 프리로드 [M2]** — player.svg, enemy.svg, ui-heart.svg 등 퍼즐 게임에서 쓰지 않는 에셋이 범용 템플릿에서 남아있음. 에러는 안 나지만 로딩 시간을 늘린다.
- **Google Fonts 외부 의존 [M1]** — "외부 에셋 0개" 기획 원칙을 위반. 오프라인이나 느린 네트워크에서 폰트 로딩 지연 가능.
- **R키 재시작 확인 UI 없음 [M3]** — 실수로 긴 게임을 날릴 수 있다. iframe에서 `confirm()` 사용 불가이므로 캔버스 기반 모달이 필요하다.
- 다음에 같은 장르를 만든다면 **이동 애니메이션을 로직과 분리한 tween 시스템**을 먼저 설계하겠다.

## 기술 하이라이트 🛠️
- **DPR(Device Pixel Ratio) 대응**: `window.devicePixelRatio`로 고해상도 디스플레이에서도 선명한 렌더링. Canvas 내부 해상도와 CSS 표시 크기를 분리하는 패턴을 적용했다.
- **동적 난이도 곡선**: 점수 구간별로 새 블록의 색상 분포와 추가 스폰 확률을 조절해 "처음 2분 여유 → 5분 후 긴장"의 난이도 커브를 코드로 자연스럽게 구현했다.
- **에셋 프리로드 + 폴백 전략**: `Promise.all` + `img.onerror = resolve`로 병렬 로딩하되, 실패해도 게임이 돌아가도록 코드 기반 폴백 렌더링을 모든 에셋에 대비시킨 방어적 설계가 인상적이다.
- **merged[][] 배열**: 한 번의 스와이프에서 한 블록이 여러 번 머지되는 버그를 원천 차단하는 단순하지만 효과적인 접근.

## 다음 사이클 제안 🚀
1. **Tween/애니메이션 시스템 공통화** — 블록 이동 슬라이드 애니메이션 미구현이 가장 큰 아쉬움이었다. 다음 게임부터는 `lerp` + `easing` 기반의 범용 tween 시스템을 먼저 구축하면 시각적 완성도가 크게 올라갈 것이다.
2. **다른 장르 도전 (arcade/action)** — 퍼즐은 검증했으니, 실시간 입력 처리와 충돌 감지가 필요한 아케이드 장르로 플랫폼의 기술적 폭을 넓혀보자.
3. **에셋 템플릿 정리** — 범용 템플릿에서 불필요한 에셋이 남아있는 문제가 반복되지 않도록, 장르별 에셋 템플릿을 분리하거나 사용하지 않는 에셋을 빌드 시 자동 제거하는 방법을 고려할 만하다.
