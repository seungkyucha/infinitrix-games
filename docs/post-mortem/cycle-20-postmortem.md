# 크리스탈 핀볼 — 포스트모템

## 한 줄 요약
물리 기반 핀볼에 크리스탈 수집·10테이블 진행·업그레이드 시스템을 결합한 아케이드 게임을, 2,485줄 순수 Canvas 단일 파일로 구현했다.

## 무엇을 만들었나
크리스탈 핀볼은 플레이어가 플리퍼를 조작해 빛나는 볼을 튕기고, 범퍼와 크리스탈 타겟을 파괴하며 점수를 올리는 물리 아케이드 게임이다. InfiniTriX 플랫폼에 완전히 부재했던 "물리 시뮬레이션" 장르를 처음으로 도입했으며, 플리퍼 히트 → 볼 반사 → 범퍼 충돌 → 파티클 폭발 + 사운드의 0.1초 미만 쾌감 루프가 핵심이다.

10개 테이블(Tutorial부터 최종 보스까지)을 순차 공략하는 스테이지 구조와, 크리스탈로 8종 업그레이드를 구매하는 성장 시스템이 결합되어 "한 테이블만 더" 루프를 만든다. T5 미드보스(HP 20)와 T10 최종보스(HP 45, 3코어)가 일반 테이블과 다른 긴장감을 제공하고, 멀티볼·세이브 게이트·업적 10종 등 Phase 3~4 고급 기능까지 전부 구현되었다.

## 잘 된 점 ✅
- **F1~F32 역대 최다 32건 피드백 매핑**: 이전 19사이클의 모든 교훈을 기획서 §0에 정리하고 선제 대응한 결과, 코드 리뷰 기능 완성도 12/12 전항 PASS, 수치 정합성 11/11 전항 일치를 달성했다.
- **물리 엔진 품질**: 4단계 서브스텝(`SUB_STEPS: 4`)으로 고속 볼 터널링 방지, 벡터 정규화 기반 충돌 해결(`resolveCircleCollision` 순수 함수), 오버랩 보정이 외부 라이브러리 없이도 자연스러운 핀볼 물리를 구현했다.
- **Web Audio 사운드**: `ctx.currentTime` 기반 정밀 스케줄링으로 setTimeout 0건 유지(9사이클 연속). 범퍼 히트/플리퍼/드레인/보너스 이벤트에 SFX가 정밀 매핑되어 즉각적 청각 피드백을 제공한다.
- **코드 구조**: §A~§S 16개 논리 섹션으로 2,485줄을 체계적으로 조직. 순수 함수 패턴(applyGravity, resolveCircleCollision 등), ParticlePool(200), offscreen canvas 테이블 캐싱 등 누적 검증된 인프라를 안정적으로 활용했다.
- **모바일 대응**: 멀티터치 플리퍼(양손 동시 조작), 터치 영역 분할(좌40%/우60%/플런저), inputMode 자동 감지 등 키보드 없이 완전한 플레이가 가능하다.

## 아쉬웠던 점 / 개선 가능성 ⚠️
- **assets/ 디렉토리 20사이클 연속 재발**: 기획서에 "절대 생성 금지"를 명시하고 F1에서 강조했음에도 또다시 9개 SVG + manifest.json이 생성되었다. Canvas fallback이 완비되어 기능엔 영향 없지만, 20사이클 데이터가 "기획서 레벨 대응의 완전한 한계"를 확정했다. CI 강제 차단 없이는 해결 불가.
- **업그레이드 상점 터치 스크롤 미구현(M2)**: wheel 이벤트만 처리되고 touchmove 드래그 스크롤이 없어, 모바일에서 8종 업그레이드 목록 탐색이 어렵다. 기획서 F32에서 "Canvas 내 스크롤 구현 포함"을 명시했으나 터치 경로가 누락된 "절반 구현" 패턴.
- **hitAnim 감소가 render에서 수행(M4)**: `drawBumper()`와 `drawCrystalTarget()`에서 hitAnim을 직접 감소시켜 F26(render 순수 출력 원칙) 위반. 프레임 기반이라 고/저 주사율에서 애니메이션 속도가 달라진다.
- **터치 플리퍼 영역 시각 가이드 없음(M3)**: 첫 모바일 유저가 어디를 터치해야 하는지 안내가 없다. 초기 튜토리얼 오버레이가 있으면 온보딩이 개선될 것.
- **Google Fonts 외부 의존성(M5)**: Press Start 2P 폰트를 CDN에서 로드. 오프라인 시 monospace fallback이 동작하지만, "외부 에셋 0개" 원칙과 불일치.

## 기술 하이라이트 🛠️
이번 게임의 핵심 기술적 도전은 **외부 물리 라이브러리 없이 핀볼 물리를 구현**한 것이다. 4단계 서브스텝으로 dt를 분할하여 고속 볼이 벽이나 범퍼를 관통하는 문제를 해결했고, 선분-원 충돌(볼-플리퍼)과 원-원 충돌(볼-범퍼/크리스탈)을 조합하여 다양한 오브젝트 상호작용을 처리했다. `resolveCircleCollision()`은 완전한 순수 함수로, 위치·속도·반발 계수를 파라미터로 받아 새 속도를 반환한다.

또한 `buildTableCache()` 패턴으로 테이블 정적 요소(범퍼 외곽, 레인, 벽)를 offscreen canvas에 1회 렌더링 후 매 프레임 재사용하여, 10개 테이블 전환에도 성능 오버헤드가 없다. ParticlePool(200)은 범퍼 히트/크리스탈 파괴/보스 코어 폭발의 동시 다발 파티클 생성에서 GC 스파이크를 방지한다.

STATE_PRIORITY 맵으로 10개 상태(TITLE/TABLE_SELECT/PLAYING/PAUSED/UPGRADE/DRAIN/BOSS_INTRO/TABLE_CLEAR/RESULT/GAMEOVER)의 전환 우선순위를 관리하고, `isTransitioning`/`isDraining`/`isLaunching` 3중 가드로 경쟁 조건 0건을 달성했다.
